# 项目结构

强度：`Advisory`

适用范围：仓库布局、模块归属、依赖方向、共享位置和配置归属的生成契约。

## 生成契约

根据观察到的仓库结构和强制执行的依赖边界编写目标规则。结构说明应有所取舍：只记录能够指导放置位置
或防止无效依赖的位置与关系，不要逐个罗列目录。

## 证据

- 仓库和工作区目录树、包清单、模块声明和导入图。
- 功能目录、共享库、配置归属、测试、资产、生成源文件、脚本、基础设施和文档位置。
- 依赖检查、lint 规则、构建目标、包边界和有代表性的导入。
- 能够确立稳定边界的归属文件和反复出现的放置模式。

## 内容

- 记录顶层区域及各自负责的职责。
- 记录功能和模块布局、放置约定及共享位置。
- 记录允许和禁止的依赖方向，以及这些方向所保护的边界。
- 如果项目中存在 UI、后端、领域、数据、基础设施、测试、资产、生成源文件、配置、脚本和文档等区域，
  记录它们之间的归属关系。
- 记录真实的强制执行机制，但其具体调用方式应留在 `Project Tools` 中。

## 边界

- 工具、运行时、构建、测试和验证命令应放在 `Project Tools` 中。
- API 契约、payload、领域词汇、生成文件编辑策略和 lint 解释应放在 `Project Rules` 中。
- 不要包含通用架构建议、臆测中的未来布局，以及名称已能自明且不施加放置或依赖约束的目录。
- 不要在多个章节或规则中重复同一项归属声明。
