# 代码设计

强度：`Default`

适用范围：跨语言生产代码的所有权、公共边界、命名、抽取、状态、依赖、控制流和注释。

## 所有权与公共边界

- 让行为靠近拥有其状态、不变量或模块的对象。仅被一个 class、service、model 或状态所有者使用的
  helper 默认属于该所有者；纯函数、无状态或更易测试都不足以成为顶层声明的理由。
- 只有当操作独立于单一所有者时才使用顶层函数或自由函数，例如共享领域逻辑、可复用规范化或解析、
  无所有者 factory、框架入口、文件级声明或被多个所有者使用的 helper。
- 公共边界由产品需求驱动。不要只为测试或调用点便利增加公共 API、测试专用 seam、抽取 helper 或
  adapter。

## 命名

- 按领域角色和可观察行为命名，不按实现细节命名。
- action 和状态转换使用精确动词。除非周边领域已使角色明确，否则避免 `handle`、`process`、
  `manager` 或 `utils` 等模糊名称。
- Boolean 名称描述其判断的条件，而不是调用方随后执行的动作。

## 抽取

- 优先保留直接、易读的调用方代码，避免 helper 间接层。
- 当 helper 命名了真实概念、隐藏有意义的复杂度、保护不变量或消除真实重复时才抽取。
- 当名称没有增加信息时，内联小型、单次使用的 helper。
- 按决策或资源边界拆分长工作流，不按任意行数拆分。具有一个清晰顺序的较长函数，优于必须一起阅读
  的分散步骤。
- 函数优先保持在约 80 行以内，但将其视为可读性指导。不要只为满足行数而抽取 helper。

## 数据与状态

- 优先使用不可变值和明确的数据形状。当命名类型或 options 对象能澄清契约时，替换松散 map、原始
  tuple、flag 集群和魔法参数。
- 可变状态应由一个所有者管理。在该所有者处明确生命周期、取消和释放路径，不要将其分散到无关
  helper。
- 当所有权在异步或 callback 边界可能变得脆弱时，在边界前缓存稳定依赖；变化状态在使用点读取。

## 依赖

- 传入单元真正需要的窄依赖，不要传入宽泛 owner、service 或全局 context。
- 依赖方向应从调用方清晰可见。当调用方可以显式提供依赖时，不要让 helper 跨层访问或从隐藏全局
  状态拉取依赖。
- 只在真实外部边界引入 adapter，不要在已经共享同一领域模型的代码之间增加 adapter。

## 控制流

- 对无效、空或已完成情况使用 early return，使主路径保持可见。
- 预期失败在本地使用 typed result、status 或 no-op return 处理。Exception 和 fatal path 只用于
  意外失败。
- 不要把重试、fallback 选择或持久化副作用隐藏在暗示简单 getter 或纯转换的名称之后。

## 注释

- 仅当注释能保存代码无法清楚表达的上下文时添加：意图、不变量、生命周期约束、外部要求、边界情况
  推理、不直观的失败处理，或为何显而易见的替代方案错误。
- 不要复述代码、描述本次编辑、重复名称，或用注释弥补本可由更清晰结构表达的问题。
- 对参数顺序、boolean flag 或隐藏所有权，优先通过名称和小型数据类型表达，不用注释解释。
