# 代码设计目标

强度：`Default`

适用范围：跨语言代码的归属、边界、清晰度、状态完整性、依赖、抽象和文档。

## 归属与 API

- 每项行为、状态、不变量和生命周期都要有一个明确的所有者。
- 所有者内部的逻辑应与所有者保持内聚；只有确实存在复用或边界时才移动逻辑。
- 公共接口应保持精简，并对应稳定的产品或领域能力。
- 不要仅为了方便测试或调用而暴露 API、移动逻辑或增加间接层。

## 清晰度与抽象

- 通过命名、类型和结构，让职责、有效状态和主要决策路径可以在局部理解。
- 在契约或控制流中清楚呈现副作用、失败方式、重试、后备行为和生命周期转换。
- 只有抽象能表达真实概念、保护不变量，或在降低维护成本的同时消除有意义的重复时，
  才引入它。
- 避免使用模糊归属，或只服务于一个简单调用点的辅助函数、适配器和通用层。

## 状态与依赖

- 从创建到清理，状态修改和生命周期转换都应保持可预期。
- 在异步工作和回调中维持正确的依赖生命周期。
- 每个代码单元只接收自身需要的能力，并明确依赖方向和跨层边界。
- 不要从隐藏的全局对象获取依赖，也不要以模糊归属的方式分散状态。

## 文档

- 注释用于保留设计原因、不变量、生命周期约束、外部要求和不直观的边界情况。
- 不要用注释复述代码、重复名称、记录修改历史或弥补不清晰的设计。
