# 代码设计目标

强度：`Default`

适用范围：跨语言代码在归属、边界、清晰度、状态完整性、依赖、抽象和文档方面的质量目标。

## 归属与边界

- 每项行为、状态、不变量和生命周期都有清晰的所有者。
- 单一所有者逻辑与其所有者保持内聚，共享逻辑的归属与实际复用范围一致。
- 公共接口表达稳定的产品或领域能力，并保持精简和封装。
- 不要仅为了方便测试或调用而新增公共 API，也不要把单一所有者逻辑提升为共享或顶层逻辑。

## 清晰度与抽象

- 命名能够表达领域角色、可观察行为、状态和条件。
- 每个代码单元职责内聚，主要决策路径可以在局部理解。
- 抽象对应真实概念、保护不变量或消除有意义的重复，并降低理解和修改成本。
- 副作用、失败方式、重试和后备行为能够从命名、契约或控制流中看出。
- 不要引入缺乏明确概念或维护价值的间接层、通用辅助函数或抽象。

## 数据与状态完整性

- 数据结构明确表达领域含义和有效状态。
- 状态变化和生命周期转换归属清晰，创建、更新、取消和释放行为可以预期。
- 异步和回调行为保持依赖生命周期有效，并区分稳定依赖与变化中的状态。
- 不要分散状态修改或生命周期归属，使有效状态或责任难以判断。

## 依赖与架构

- 每个代码单元只接收自身需要的能力，依赖方向保持明确。
- 层之间的交互遵守领域边界，每项跨层依赖都有明确意图且清晰可见。
- 公共 API 和适配器对应真实的产品、领域或外部边界，并带来明确的维护价值。
- 不要从隐藏的全局对象中拉取依赖，也不要在没有真实边界的地方增加适配器。

## 文档

- 命名、类型和结构直接表达代码行为。
- 注释保留代码难以清楚表达的设计原因、不变量、生命周期约束、外部要求和边界情况。
- 不要用注释复述代码、记录修改历史、重复名称或弥补不清晰的设计。
