# 代码设计

强度：`Default`

适用范围：跨语言的生产代码归属、公共边界、命名、提取、状态、依赖、控制流和注释。

## 归属与公共边界

- 行为应靠近拥有相应状态、不变量或模块的主体。默认情况下，只被一个类、服务、模型或状态所有者
  使用的辅助逻辑应归该所有者所有；仅仅因为它纯粹、无状态或更易测试，不足以将其提升到顶层。
- 只有当操作不依赖任何单一所有者时，才使用顶层函数或自由函数，例如共享领域逻辑、可复用的规范化
  或解析逻辑、无所有者的工厂、框架入口点、文件级声明，或供多个所有者使用的辅助函数。
- 公共边界应由产品需求决定。不要仅为方便测试或调用而新增公共 API、测试专用接缝、提取辅助函数
  或适配器。

## 命名

- 根据代码的领域角色和可观察行为命名，不要根据实现细节命名。
- 动作和状态转换使用准确的动词。除非上下文已使其领域角色足够明确，否则避免使用 `handle`、
  `process`、`manager` 或 `utils` 等含糊名称。
- 布尔值按其判断的条件命名，不要按调用者随后执行的动作命名。

## 提取

- 优先保留直接、易读的调用方代码，避免不必要的辅助函数间接层。
- 当辅助函数能够命名真实概念、隐藏实质复杂度、保护不变量或消除真实重复时，再将其提取出来。
- 小型且只使用一次的辅助函数，如果名称没有增加信息，应直接内联。
- 应在决策边界或资源边界处分割长工作流，而不是按任意行数切分。与其把必须连在一起阅读的步骤拆散，
  不如保留一个顺序清晰的较长函数。
- 函数最好大致控制在 80 行以内，但这只是可读性指导。不要仅为满足行数而提取辅助函数。

## 数据与状态

- 优先使用不可变值和明确的数据形状。如果命名类型或选项对象能让契约更清楚，就用它们替代松散映射、
  原始元组、成组标志位和魔法参数。
- 可变状态只能由一个主体管理。生命周期、取消和释放路径应在该所有者处明确表达，
  不要散落到互不相关的辅助函数中。
- 如果跨越异步或回调边界后归属关系可能变得脆弱，应提前缓存稳定依赖；变化中的状态则在使用时读取。

## 依赖

- 传入当前单元真正需要的窄依赖，不要传入宽泛的所有者、服务或全局上下文。
- 依赖方向应从调用方清晰可见。如果调用方可以显式提供依赖，不要让辅助函数跨层访问，
  也不要从隐藏的全局对象中拉取依赖。
- 只在真实的外部边界引入适配器，不要在已经共享同一领域模型的代码之间添加适配层。

## 控制流

- 对无效、空值或已完成的情况使用提前返回，让主路径保持清晰。
- 对预期失败就地使用类型化结果、状态或空操作返回处理。异常和致命路径只用于意外失败。
- 不要把重试、后备方案选择或持久化副作用隐藏在听起来像简单 getter 或纯转换的名称之后。

## 注释

- 只有在需要保留代码无法清楚表达的上下文时才添加注释：意图、不变量、生命周期约束、外部要求、
  边界情况推理、非显而易见的失败处理，或看似合理的替代方案为何不可取。
- 不要复述代码、描述本次修改、重复名称，也不要用注释弥补本可通过更清晰结构表达的内容。
- 对参数顺序、布尔标志或隐藏归属问题，优先使用清晰命名和小型数据类型，而不是依靠注释说明。
