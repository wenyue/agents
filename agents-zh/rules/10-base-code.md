# 代码品味

强度：`Default`

适用范围：跨语言的生产代码设计与 review 默认规则。

## 所有权

- 行为应靠近拥有其状态、不变量或模块的对象。仅被一个 class、service、model 或状态所有者使用的 helper，默认属于该所有者。无状态、纯函数或便于测试不足以成为顶层函数的理由。
- 只有当操作不依赖单一所有者时才使用顶层函数或自由函数，例如共享领域逻辑、可复用规范化或解析、没有明确所有者的 factory、框架要求的入口、文件级声明或被多个所有者使用的 helper。
- 公共边界由产品需求驱动。不要只为测试或调用方便增加公共 API、测试专用 seam、抽取 helper 或 adapter。

## 命名

- 按领域角色和可观察行为命名，不按实现细节命名。
- action 和状态转换使用精确动词。除非上下文已明确领域含义，否则避免 `handle`、`process`、`manager`、`utils` 等模糊名称。
- Boolean 名称应描述被判断的条件，而不是随后要执行的动作。

## 抽取

- 优先保留直接且易读的调用方代码。只有当 helper 命名了真实概念、隐藏有意义的复杂度、保护不变量或消除真实重复时才抽取。
- 如果名称没有增加信息，内联非常小且只使用一次的 helper。
- 按决策或资源边界拆分长工作流，而不是按任意行数拆分。顺序清晰的长函数优于必须跨处阅读的零散步骤。
- 函数通常控制在约 80 行以内，但这是可读性指导而不是硬限制。不要只为满足行数而抽取 helper。

## 数据与状态

- 优先使用不可变值和明确的数据形状。当命名类型或 options 对象能让契约更清晰时，避免松散 map、原始 tuple、flag 集群或魔法参数。
- 可变状态应由一个所有者管理。生命周期、取消和释放路径应在该所有者处明确，不要分散到无关 helper 中。
- 当语言或框架使异步/回调边界上的所有权脆弱时，在边界前缓存稳定依赖；会变化的状态则在使用时读取。

## 依赖

- 传入单元真正需要的窄依赖，不要传入宽泛的 owner、service 或全局 context。
- 依赖方向应从调用方清晰可见。当调用方可以显式提供依赖时，避免 helper 跨层访问或从隐藏全局状态取依赖。
- 只在真实外部边界引入 adapter，不要在已经共享同一领域模型的代码之间增加 adapter。

## 控制流

- 对无效、空或已完成情况使用 early return，使主路径保持可见。
- 预期失败在本地使用类型化 result、status 或 no-op return 处理。Exception 或 fatal path 只用于真正意外的失败。
- 不要把重试、fallback 选择或持久化副作用隐藏在听起来像简单 getter 或纯转换的名称之后。

## 注释

- 仅在注释能保存代码难以清楚表达的上下文时添加，例如意图、不变量、生命周期约束、外部要求、边界情况推理、非显然失败处理或为何不能采用显而易见的方案。
- 不要复述代码、描述本次编辑、重复名称，或用注释弥补本应通过结构解决的不清晰。
- 对不清晰的参数顺序、boolean flag 或隐藏所有权，优先通过名称和小型数据类型表达。
