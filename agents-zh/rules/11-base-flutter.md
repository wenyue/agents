# Dart 与 Flutter 指引

强度：`Default`

适用范围：Dart 与 Flutter 的归属、状态、生命周期、UI、路由、模型和分析边界。

## 公共接口与归属

- 公共 API 应保持精简并由产品需求驱动；不要为了测试或方便调用而暴露辅助函数或移动
  所有者内部行为。
- 领域不变量应留在负责它的值类型或服务中。
- 所有者内部行为默认使用实例成员。
- 顶层函数只用于框架入口、文件级声明、共享算法，或确实没有明确所有者的逻辑。

## 数据形态

- 优先使用不可变、精确类型的数据，并用命名类型表达有意义的状态。
- record 只用于小型局部元组返回；如果命名类型能提高清晰度，就使用命名类型。

## 状态管理

- 共享、功能级和跨组件状态使用 Riverpod；简单的局部事项留在负责它的组件中。
- 如果应用已经使用代码生成，就使用生成的 provider。
- build 期间观察响应式状态，事件处理程序中读取状态，只用监听处理副作用。
- 只关心状态对象的一部分时，将 `select` 与 `ref.watch` 或 `ref.listen` 配合使用。
- 生命周期不应依赖单个页面的服务和仓库使用 `keepAlive: true`。

## Provider 生命周期

- provider 的 `build()` 是响应式的，依赖变化时可能再次运行。
- 创建可释放资源后立即注册对应的 `onDispose` 回调。
- 必须在任何 `await` 之前注册释放逻辑。
- 不要在 `onDispose` 中给 `state` 赋值、读取 provider 或访问 `Ref`。

## 异步边界

- 经过异步间隔后，使用受生命周期约束的状态前，应确认负责它的组件或 provider 仍处于
  mounted 状态。
- 异步间隔后仍需使用的稳定依赖，应在间隔前解析。
- 会变化的 provider 状态应在异步间隔和 mounted 检查之后读取。

## 组件

- 选择能够负责所需生命周期的最简单组件类型。
- 展示型组件应保持小而稳定；只有交互确实需要时才引入 controller 生命周期。
- 添加替代方案前，先使用应用已经建立的主题和 UI 抽象。

## 路由

- 页面使用应用已经建立的 router 和生成路由抽象。
- 由 Navigator 管理的对话框、底部面板和浮层使用
  `Navigator.of(context).pop(result)`。

## 数据模型

- 使用应用已经建立的序列化和不可变模型抽象。
- 如果存储细节与应用逻辑不同，应将持久化模型与领域模型分开。

## 分析

- 机械样式、命名、导入、类型、注释和弃用检查遵循目标仓库的 formatter、analyzer 和 lint
  配置。
- 代码、标识符和注释使用英语。
- 当前时间会影响行为时，使用可注入或可测试的时间源。
