# Go 指引

强度：`Default`

适用范围：工具配置之外的 Go 包归属、API 形态、错误、context、并发和日志约束。

## 代码形态

- 代码应直接、作用域应小，并保持浅平的成功路径。
- 公共 API 应由产品需求驱动。不要仅为测试或臆测中的复用而导出名称、添加接口或引入接缝。
- 只有辅助函数或私有类型能命名真实概念、理清数据流或消除实质复杂度时，才进行提取。
- 不要添加 `init()` 函数。应从构造函数、`main` 或包自有的初始化函数中显式初始化。

## 错误

- 检查每个返回的错误；不要用空白标识符丢弃错误。
- 类型断言可能失败时使用 `value, ok := x.(T)`；只有调用点能够直接看出类型由不变量保证时，
  才使用 `x.(T)`。
- 错误跨越包边界时，使用 `%w` 包装并提供有用上下文；通过 `errors.Is` 或 `errors.As`
  检查。
- 新建动态错误前，优先复用已有的哨兵错误。
- 除非处于无法继续运行的顶层启动路径，否则避免使用 `panic`、`logger.Fatal` 和
  `os.Exit`。

## Context 与并发

- 符合 Go 常规模式时，将 `context.Context` 放在首个参数，并在循环和长时间运行的工作中
  响应取消。
- 使用归属包的同步原语保护共享可变状态。
- 包级全局变量应有明确用途、保持稳定，并在一个位置初始化。

## 日志

- 遵循归属包的日志约定。不要引入 `fmt.Print*` 或标准库 `log` 来输出运行时信息。
- 日志消息应指出操作，并包含诊断失败所需的值。

## 工具归属

- 布局、命名、限制、导入、注释和 suppression 语法交给仓库的 formatter 和 linter。
- 配置阈值应在负责它的配置中修改，不要在本规则中重复。
