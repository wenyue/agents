# Go 指引

强度：`Default`

适用范围：Go 的格式、代码形态、命名、错误、注释、并发、集合和日志默认约定。

## 代码形态

- 编写紧凑、完善、易读的 Go 代码。直接表达意图，缩小作用域，避免重复分支；只有当辅助函数能够
  命名真实概念或消除实质复杂度时，才提取它。
- 对无效输入、取消和错误路径优先使用提前返回，使成功路径保持浅平。
- 公共 API 应由产品需求决定。不要仅为测试或臆测中的复用而导出名称、添加接口或引入接缝。
- 小型局部结构体或私有辅助类型能让数据流更清晰时，可以使用；不要为单个调用点引入宽泛的包级抽象。
- 不要添加 `init()` 函数。应从构造函数、`main` 或包自身的初始化函数中显式完成初始化。

## 格式与限制

- 使用 `golangci-lint-v2 fmt` 格式化。不要手工调整导入分组或换行来对抗 `gci`、`gofmt`、
  `gofumpt`、`goimports` 或 `golines`。
- 保留 `wsl_v5`、`nlreturn` 和 `whitespace` 要求的空行；可读性意味着区块清晰，而不是代码挤成一团。
- 每行不超过 100 个字符，长调用和字符串拼接交给 `golines` 换行。
- 导入按标准库、第三方、本地模块分组。各组之间空一行，并由格式化工具排序。
- 函数应低于 `funlen` 的 80 行、50 条语句限制，并低于 `cyclop` 16、`gocyclo` 20 和 `nestif` 8。
- 当辅助函数能拉平控制流、消除重复或命名一个领域步骤时，再进行提取。极小的单次使用辅助函数
  如果让直白代码反而更难理解，应将其内联。

## 命名

- 导出标识符使用 `PascalCase`，未导出标识符使用 `camelCase`。
- 缩写形式应保持一致：`ID`、`CID`、`IPFS`、`IPNS`、`DHT`、`URL`、`UDP` 和 `TCP`。
- 哨兵错误命名为 `ErrX`，错误类型名称以 `Error` 结尾。
- 短名称只用于紧凑作用域，或 lint 允许的名称：`err`、`ok`、`id`、`i`、`j`、`k`、`v`、
  `db` 和 `to`。作用域更大的值应按其角色命名。
- 可行时将布尔值命名为谓词，例如 `ok`、`enabled`、`hasRoute` 或 `shouldRetry`。

## 错误

- 所有返回错误都必须检查；空白赋值和类型断言也不能跳过检查。
- 错误跨越包边界时，应使用 `%w` 包装并提供有用上下文；通过 `errors.Is` 或 `errors.As` 检查。
- 新建动态错误前，优先复用已有的哨兵错误。
- 类型断言使用 comma-ok 形式。生产代码中不要直接使用 `x.(T)`。
- 除非处于无法继续运行的顶层启动路径，否则避免使用 `panic`、`logger.Fatal` 和 `os.Exit`。

## 注释

- 注释用于说明意图、不变量、并发、重试行为、安全取舍或不直观的领域规则。不要复述下一条语句。
- 为导出声明添加文档注释。每条注释都以相应标识符开头，并以标点结尾。
- 在较长的多步骤函数中，如果能让阅读路径更清晰，可以使用简短的阶段注释。
- 注释使用英语，首字母大写并加结尾标点，避免触发 `godot`、`revive` 和 `misspell`。
- `TODO(name): ...` 和 `FIXME(name): ...` 必须包含具体的后续动作。
- 每个 `//nolint:<linter>` 都必须附上具体原因。应优先修改代码以满足 lint。

## Context 与并发

- 符合 Go 常规模式时，将 `context.Context` 放在首个参数，并在循环和长时间运行的工作中响应取消。
- 除非代码说明了生命周期为何正确，否则不要在循环或闭包中创建嵌套 context。
- 使用归属包的同步原语保护共享可变状态。
- 包级全局变量应有明确用途、保持稳定，并在一个位置初始化。

## 数据与集合

- 固定值使用 `const`，可调数字使用命名常量。不要在控制流中放置魔法数字。
- 如果切片和映射的大小已知或可低成本获得，应预先分配容量。
- 当需要明确表达顺序和唯一性时，使用 `map[T]struct{}` 或小型辅助函数去重。
- 结构体标签应保持一致，其格式和命名交由 `tagalign` 和 `tagliatelle` 决定。

## 日志

- 遵循归属包的日志约定。不要引入 `fmt.Print*` 或标准库 `log` 来输出运行时信息。
- 日志消息应指出操作，并包含诊断失败所需的值。
- 消息中包含值时，优先使用格式化日志方法。
