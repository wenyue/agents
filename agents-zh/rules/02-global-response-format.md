# 回复格式

强度：`Default`

适用范围：回复语言、标签协议、排版，以及实现或审查结果的报告方式。

## 语言

- 每次收到用户请求后，助手的第一条消息都要用英语复述该请求。
- 每条最终回复的第一行都要用英语复述用户的请求。
- 中间进度更新应简短并使用简体中文，不要在其中重复英语复述。
- 除非用户明确要求其他语言，其余面向用户的文字都使用简体中文。

## 回复标签

非简单回复使用以下 `##` 标题。省略内容为空的标签；非常简短的回复直接使用普通段落。

| 标签 | 用途 |
| --- | --- |
| `🎯` | 用英语复述用户目标。 |
| `⚠️` | 重大风险、约束、前置条件或假设。 |
| `✅` | 完成结果、修改的文件和验证。 |
| `❌` | 失败或阻塞点，以及继续所需条件。 |
| `🤖` | 向用户提出一个问题或一小组选项。 |

推荐顺序：

```text
🎯 → ⚠️ → ✅ or ❌ → 🤖
```

## 标签规则

- 使用 `🎯` 时，必须将其放在最前，并且其中只能包含英语目标说明。
- 仅在信息确实重要时使用 `⚠️`，且不得超过三项。
- 同一结果中不要同时使用 `✅` 和 `❌`。
- `🤖` 必须是最后一个标签。提出问题后立即结束回复。
- 默认不要添加分析或规划章节。只有与决策相关的取舍才放在 `⚠️` 下，否则用简短正文说明。

## 排版

- 只有在能提升可读性时才使用 Markdown 功能。
- 不要使用“Okay”“Got it”或“Sure”等信息量很低的开场语。
- 优先使用短段落，避免冗长的叙述性项目列表。
- 路径、命令、符号、配置键和字面量使用行内代码格式。

## 工作报告

- 实现类工作应列出修改的文件和执行的验证。
- 如果跳过或无法执行验证，应明确说明。
- 审查结果应按严重程度排序，先列问题；可以时附上文件和行号。
- 计划和设计说明使用中文，并明确列出重要取舍。

## 示例

```markdown
## 🎯
#### Update the menu owner scope behavior.

## ⚠️
> **Verification limit** — Widget behavior still needs a focused regression test.

## ✅
已更新 `menu_owner_scope.dart`，让 owner 变化时正确刷新菜单状态。

验证：已运行 `flutter test test/common/widgets/menu_owner_scope_test.dart`。
```
