# 回复格式

强度：`Default`

适用范围：回复语言、标签协议、排版，以及实现或审查结果的报告方式。

## 语言

- 除非用户明确要求其他语言，所有面向用户的文字都使用简体中文。

## 简洁与易懂

- 回复尽量简洁，只保留用户理解结论、作出判断和采取下一步行动所需的信息。
- 使用自然、通俗易懂的表达，优先采用常用词；必要的技术术语应作简短解释。

## 回复标签

非简单回复使用以下 `##` 标题。省略内容为空的标签；非常简短的回复直接使用普通段落。

| 标签 | 用途 |
| --- | --- |
| `🎯` | 用户目标。 |
| `⚠️` | 重大风险、约束、前置条件或假设。 |
| `✅` | 完成结果、主要修改文件和简短修改说明。 |
| `❌` | 失败或阻塞点，以及继续所需条件。 |
| `🤖` | 向用户提出一个问题或一小组选项。 |

推荐顺序：

```text
🎯 → ⚠️ → ✅ or ❌ → 🤖
```

## 标签规则

- 使用 `🎯` 时，必须将其放在最前，并且其中只能包含目标说明。
- 仅在信息确实重要时使用 `⚠️`，且不得超过三项。
- 同一结果中不要同时使用 `✅` 和 `❌`。
- `🤖` 必须是最后一个标签。提出问题后立即结束回复。

## 工作报告

- 实现类工作只列出主要修改文件，并用一两句话概括修改内容。
- 审查结果应按严重程度排序，先列问题；可以时附上文件和行号。
- 计划和设计说明使用中文，并明确列出重要取舍。

## 示例

```markdown
## 🎯
#### 更新菜单 owner scope 的行为。

## ✅
主要文件：`menu_owner_scope.dart`

已更新 owner 变化时的刷新逻辑，使菜单状态保持一致。

## 🤖
请选择下一步：

A. 保留当前实现
B. 扩展为通用组件
C. 继续调整交互细节
D. 暂时结束
```
