# 响应格式

强度：`Default`

适用范围：响应语言、标签协议、格式，以及实现或 review 报告。

## 语言

- 每次用户请求后的第一条助手消息用英文重述请求。
- 每个最终回复的第一行用英文重述用户请求。
- 中间进度更新使用简体中文并保持简洁；不要在这些更新中重复英文重述。
- 除非用户明确要求其他语言，其余面向用户的文本均使用简体中文。

## 响应标签

非简单回复使用以下 `##` 标题。省略空标签；非常简短的回复使用普通正文。

| 标签 | 用途 |
| --- | --- |
| `🎯` | 用英文重述用户目标。 |
| `⚠️` | 实质风险、约束、前置条件或假设。 |
| `✅` | 已完成结果、修改文件和验证。 |
| `❌` | 失败或 blocker，以及继续所需条件。 |
| `🤖` | 一个用户问题或少量选项。 |

推荐顺序：

```text
🎯 → ⚠️ → ✅ or ❌ → 🤖
```

## 标签规则

- 使用 `🎯` 时必须放在最前面，且只包含英文目标文本。
- 仅在存在实质信息时使用 `⚠️`，并限制为三项以内。
- 不要在同一结果中同时使用 `✅` 和 `❌`。
- `🤖` 必须位于最后；提出问题后停止回复。
- 默认不要添加分析或规划章节。只把影响决策的权衡放入 `⚠️` 或简洁正文。

## 格式

- 仅在 Markdown 功能能改善可浏览性时使用它们。
- 不要使用“好的”“明白了”“当然”等低信息开场。
- 叙述内容优先使用短段落，不使用冗长的项目列表。
- 路径、命令、symbol、配置 key 和字面量使用行内代码。

## 工作报告

- 实现工作应说明修改文件和执行的验证。
- 如果验证被跳过或阻塞，应明确说明。
- Review 应先列 findings，按严重程度排序，并尽可能提供文件和行号。
- 计划和设计说明使用中文，并明确实质权衡。

## 示例

```markdown
## 🎯
#### Update the menu owner scope behavior.

## ⚠️
> **验证限制** — Widget 行为仍需要一个聚焦的回归测试。

## ✅
已更新 `menu_owner_scope.dart`，让 owner 变化时正确刷新菜单状态。

验证：已运行 `flutter test test/common/widgets/menu_owner_scope_test.dart`。
```
