# 重命名

在已批准的兼容边界内，完整重命名一个明确的概念。尽可能依据语义引用确定范围；除非用户批准更改，否则保留对外名称。

## 前提条件

1. 确认目标当前的名称、种类、声明、作用域和预期的新名称。
2. 阅读目标项目关于命名、生成文件、API 和验证的规则。
3. 判断该名称是私有、公共、用于序列化、由工具生成、持久化、通过反射使用，还是被仓库外部使用。
4. 只有符号身份、预期范围或公共兼容性决策不明确时才询问。

## 确定重命名范围

优先使用理解 import、继承、分派和语言语义的项目级符号或引用工具。如果没有此类工具，将全仓库全词搜索与调用点、类型和配置检查结合使用。

编辑前对每处匹配进行分类：

- 标识该符号的声明、import、引用、override、测试和文件名；
- 指代同一概念的注释和用户可见文本；
- 基于字符串的反射、注册、route 或动态查找；
- 生成输出及其源头；
- 序列化键、协议字段、数据库列、配置键和其他外部契约；
- 位于其他作用域、与目标无关的同名符号。

文本匹配本身不能证明它属于本次重命名。

## 兼容边界

- 有证据表明不存在外部使用方时，完整重命名私有符号和仓库内部符号。
- 对公共 API 或外部契约，如果请求尚未给出兼容性决策，编辑前先说明影响并取得决定。
- 只有确实需要兼容，且项目支持迁移路径时，才添加 deprecated alias。不得默认创建 alias。
- 序列化、协议、持久化和配置名称必须保持不变，除非它们明确属于已批准的重命名范围。
- 修改生成文件的源头并重新生成；绝不手工编辑生成输出。

## 工作流程

1. 记录已批准的新旧名称映射和兼容策略。
2. 更新声明和每处已确认的语义引用。
3. 只有文件、测试、注释、文档或镜像文本确实标识同一概念时，才对其重命名。
4. 按需重新生成由项目拥有的输出。
5. 再次搜索旧名称，将每处剩余匹配归类为预期保留或遗漏。
6. 运行目标项目的格式化、静态检查和直接受影响的测试。

## 停止条件

- 如果同一名称可能对应多个符号，且无法证明目标作用域，立即停止。
- 在改变尚未批准的公共契约或外部契约前停止。
- 如果必须重新生成，但无法确定其源头或生成命令，立即停止。

## 结果

报告被重命名的概念、兼容性结果、受影响范围、有意保留的旧名称、重新生成的输出和验证结果。
