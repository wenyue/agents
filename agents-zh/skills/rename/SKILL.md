# 重命名

在批准的兼容边界内完整重命名一个已识别概念。尽可能使用语义引用证据，并保留外部名称，除非用户批准改变它们。

## 前置条件

1. 识别目标的当前名称、类型、声明、作用域和预期新名称。
2. 阅读目标项目的命名、生成文件、API 和验证规则。
3. 确定名称是私有、公共、序列化、生成、持久化、反射使用，还是被仓库外消费。
4. 只有符号身份、预期范围或公共兼容决策不明确时才询问。

## 查找重命名 Surface

优先使用能理解 import、继承、dispatch 和语言语义的项目感知符号或引用工具。如果不可用，将全仓 whole-word 搜索与调用点、类型和配置检查结合使用。

编辑前对每个匹配项分类：

- 标识该符号的声明、import、引用、override、测试和文件名；
- 指向同一概念的注释和用户可见文本；
- 基于字符串的反射、注册、route 或动态查找；
- 生成输出及其事实源；
- 序列化键、协议字段、数据库列、配置键和其他外部契约；
- 不同作用域内无关的同名符号。

不要把文本匹配视为它属于本次重命名的证明。

## 兼容边界

- 当证据表明没有外部消费者时，完整重命名私有和仓库内部符号。
- 对公共 API 或外部契约，在请求没有给出兼容决策时，编辑前说明影响并取得该决策。
- 只有需要兼容且项目支持迁移路径时才添加 deprecated alias。不要默认创建 alias。
- 除非序列化、协议、持久化和配置名称明确位于批准的重命名范围内，否则保持不变。
- 编辑生成文件的事实源并重新生成；绝不手工编辑生成输出。

## 工作流

1. 记录批准的旧名称到新名称映射和兼容策略。
2. 更新声明和每个确认的语义引用。
3. 只有文件、测试、注释、文档或镜像文本标识同一概念时才重命名它们。
4. 需要时重新生成自有输出。
5. 再次搜索旧名称，并将每个剩余匹配分类为预期残留或遗漏。
6. 运行目标项目的 formatter、静态检查和直接受影响测试。

## 停止条件

- 同一名称解析为多个合理符号且无法证明范围时停止。
- 改变未批准的公共或外部契约前停止。
- 需要重新生成但无法识别其事实源或命令时停止。

## 结果

报告重命名概念、兼容结果、受影响 surface、有意保留的旧名称、重新生成的输出和验证结果。
