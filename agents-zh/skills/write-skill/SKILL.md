# 编写 Skill

创建一个连贯的 Skill 契约，使另一个 Agent 无需隐藏上下文即可执行。编写前先确定其分发范围和契约
角色；每次调用都对整个 Skill 的质量负责，即使请求只指出一个缺陷。

## 全局重写不变量

- 每次创建或更新都从 Skill 当前的完整目标出发重新综合。
- 把现有 Skill 及其资源视为证据和遗漏检查表，而不是不可变骨架或首选大纲。
- 在决定保留哪些现有表述前，重新建立触发条件、结果、分发范围、所有权、边界、工作流、失败行为、
  验证和资源。
- 生成完整候选并连贯替换拥有的产物。Diff 只是交付机制；绝不要为了最小文本补丁而优化 Skill。
- 把每项有效要求整合到其自然 owner 中。删除已取代、重复、矛盾、错位以及记录修改历史的内容。
- 不要仅为处理最新请求而追加异常、备注、附录或尾部章节。重写控制该行为的章节，然后再次审查整个 Skill。
- 当当前意图和证据仍支持时保留 Skill 的既定方向，但不要仅因弱结构或陈旧表述已经存在就保留它。

## 编写前分类

先确定所有权范围。共享产物分为两类：直接执行自身工作流的共享 Skill，以及编写完整目标自有 Skill
的共享 Skill 生成契约。使用清单、公共目录、同步所有权、安装范围和请求输出作为证据。

| 类别 | 目标和证据 | 允许的具体程度 | 必需边界 |
| --- | --- | --- | --- |
| 项目私有 Skill | 服务于一个目标仓库并由其拥有，无论手写还是生成 | 当前仓库根相对路径、真实命令、服务、模块和项目规则 | 优先保证目标准确性；不要发布或泛化不受支持的事实 |
| 共享 Skill | 跨仓库分发并直接执行自身工作流 | 跨仓库不变量、Skill 自有相对资源和稳定的协议自有路径 | 运行时解析项目事实，并把目标专用策略交给目标规则 |
| 共享 Skill 生成契约 | 跨仓库分发，根据目标证据编写完整的目标自有 Skill | 语义化证据类别和稳定的协议自有路径 | 描述编写过程和生成契约；不要表现得像最终目标 Skill |

项目私有 Skill 通常包含最终可执行的项目流程。如果明确需要项目私有 Skill 生成契约，使用生成契约
结构，但证据和输出仍由该仓库拥有。不要仅通过删除具体路径把目标专用 Skill 变成共享 Skill；应围绕
真正稳定的跨仓库契约重新设计，否则保持私有。

## 证据

读取拥有的完整 Skill 目录，并收集适合其类别的证据。

共同证据：

- 用户要求的结果、示例、约束和兼容要求；
- 现有 `SKILL.md`、脚本、参考、资产、镜像、测试和真实使用产物；
- 适用的所有权规则、清单、验证器、wrapper、公共目录和权威规范。

类别专用证据：

- 对项目私有 Skill，检查当前项目规则、代码、配置、命令、生成文件所有权、服务和验证行为。
  这些事实优先于通用最佳实践。
- 对共享 Skill，识别在受支持仓库和平台中持续正确的最小行为。把单一仓库事实视为示例而非默认值。
- 对共享 Skill 生成契约，检查其生成 Skill 必须消费的目标证据类别，并用存在实质差异的代表性仓库测试契约。

把稳定执行要求与实现历史、先前编辑、个人偏好以及应属于项目文档的说明分开。

## 路径和所有权规则

- 绝不要在 tracked Skill 或资源中硬编码绝对文件系统路径。不得嵌入盘符、用户主目录、checkout
  位置或机器专用临时目录。
- 当工具运行时需要绝对路径时，从 Skill 根目录、仓库根目录、任务输入或其他已发现 owner 推导。
  绝不要把解析后的机器路径持久化为编写内容。
- 使用相对于 Skill 根目录的路径引用 Skill 自有脚本、参考和资产。参考只保持一层，并明确说明何时读取。
- 在项目私有 Skill 中，项目文件使用仓库根相对路径；只有当前项目证据证明时才记录具体命令。
- 在共享 Skill 中，使用语义化项目目标和运行时发现。只有共享协议在所有受支持目标中拥有并标准化
  某路径时，才能保留该具体路径。
- 在共享 Skill 生成契约中，除非路径属于共享生成协议本身，否则用语义方式描述目标文件。要求生成的
  项目 Skill 从目标证据解析具体路径。
- 通用 Skill 编写约束属于 `write-skill`。目标事实和行为策略属于目标仓库规则；应引用而不是复制。
- 每项指令只保留一个事实源。除非平台需要额外运行时字段，wrapper 只包含平台元数据或事实源引用。

## 必需格式和契约结构

每个 Skill 都应具备：

1. 按本项目约定，仅包含 `name` 和 `description` 的 YAML frontmatter。
2. 不超过 64 个字符、与目录匹配的小写连字符名称。
3. 同时说明 Skill 做什么以及哪些具体场景会触发它的描述。所有触发信息都放在这里，因为正文只在
   激活后加载。
4. 一个 H1 标题，随后是简短的结果和边界说明。
5. 祈使式指令、明确所有权、基于证据的决策、验证，以及定义好的结果或交付。

分类后选择契约结构。只有职责证明必要时才增删章节；不要混合多套完整模板。

| 契约结构 | 必需章节顺序 | 额外约束 |
| --- | --- | --- |
| 项目私有操作型或诊断型 | 前置条件或证据；工作流或阶段；停止条件；验证；结果 | 使用已验证目标事实，并在目标任务结果处结束 |
| 项目私有编排器 | 所有权；托管资产；工作流；存在候选交付时的审查或验收门禁；验证；输出 | 协调目标自有组件，不吸收其实现契约 |
| 共享操作型或诊断型 Skill | 前置条件或证据；工作流或阶段；停止条件；验证；结果 | 只保留跨仓库行为，并在变更前解析本地事实 |
| 共享编排型 Skill | 所有权；托管资产；协调工作流；审查门禁；验收门禁；验证；输出 | 让编排、审查和验收与生成或委派的实现保持分离 |
| 共享 Skill 生成契约 | 证据；编写工作流；生成 Skill 契约；审查和交付 | 生成完整的目标自有候选；绝不混合编写指令和生成后的运行时流程 |

## 工作流

1. 把产物分类为项目私有 Skill、共享 Skill 或共享 Skill 生成契约。起草前记录该决策的证据。
2. 定义 Skill 必须完成的完整工作及必须排除的职责。从证据推断清晰细节；只有未解决的歧义会实质
   改变行为时才询问。
3. 盘点每项现有声明和资源。将其分类为保留、重写、删除或移动到实际 owner。
4. 从头陈述 Skill 的不变量、完成条件、停止条件和跨 Skill 边界，再考虑复用当前标题或句子。
5. 选择契约结构，并按必需顺序勾勒完整候选。
6. 为精确触发编写 frontmatter。除非目标运行时明确要求并支持，否则删除非标准元数据。
7. 把正文写成一个端到端契约。将保留事实整合到自然章节，而不是保留旧文档顺序。
8. 通过渐进披露设计资源。核心决策保留在 `SKILL.md`；只有条件性或详细材料才移入直接引用的资源。
9. 在同一连贯变更中更新所有拥有的资源、语言镜像、公共清单、wrapper 和契约测试。保留无关项目
   自有文件和既有 Git 状态。
10. 不查看 diff，直接通读完整候选并应用防退化和验证门禁。任一门禁失败时再次重写。

## 内容和资源

- 只包含另一个有能力的 Agent 为可靠完成任务所需的非显然流程知识、领域约束和决策。
- 设计一个连贯工作单元。拆分无关职责，不要让一个 Skill 无限扩张；但不要把总是一起加载的单一
  端到端工作切成多个 Skill。
- 让具体程度匹配脆弱性：上下文变化时解释目标和原因；变更有风险时使用精确顺序；确定性关键时
  使用可执行脚本。
- 保持 `SKILL.md` 简洁，让每条指令都值得占用注意力。核心工作流变得难以扫描前，将条件性细节
  移到参考；除非目标运行时证明不同限制，否则主文件保持在 500 行以内。
- 只为重复确定性行为或脆弱操作添加脚本，并使用安全代表性输入测试。明确其依赖、错误和恢复行为。
- 只有正文说明读取条件时才添加参考。只在 Skill 直接用其生成输出时添加资产。
- 仅在能消除行为歧义时使用简洁示例。不要解释 Agent 已知的概念、叙述显然动作或描述产生指令的编辑。
- 除非外部打包标准明确要求，否则不要在 Skill 内创建 README、changelog、安装或快速参考文件。

## 防退化门禁

出现任一条件时拒绝候选并重新编写：

- 最新请求以追加备注出现，而不是改变控制该行为的契约；
- 同一义务在多个章节重复，或与旧异常冲突；
- 项目私有事实泄漏进共享契约，或本地 Skill 以可移植性为名退化成模糊通用建议；
- 共享 Skill 生成契约包含最终目标流程，或共享 Skill 包含另一个 Skill 的编写指令；
- 仍保留过时名称、路径、假设、占位符或废弃工作阶段；
- 文档增长却没有新增独立职责或必要证据；
- 旧章节顺序仅因要缩小 diff 而保留；
- 内容描述变更历史、审查讨论或最近修复了什么；
- 资源与正文重复，或没有任何明确指令会访问它。

验收前，必须对以下所有问题回答“是”：

- 如果不存在旧版本，今天是否仍会写出这个 Skill？
- 其项目私有或共享所有权是否明确，并得到公共目录或仓库证据支持？
- 如果共享，它属于共享 Skill 还是共享 Skill 生成契约是否无歧义？
- 每个段落是否都拥有唯一、当前、可执行的信息？
- Agent 是否能识别开始条件、完成条件、停止条件和交付？
- 除非职责确实扩展，这次重写是否降低或保持了复杂度？

## 验证和交付

1. 运行当前 Skill 验证器，以及覆盖清单、wrapper、镜像、同步和所有权的全部项目契约测试。
2. 对项目私有 Skill，针对目标仓库的真实配置和验证 surface 执行修改后的脚本及代表性工作流。
3. 对共享 Skill，在能覆盖运行时发现、目标规则优先级和停止条件的代表性上下文中正向测试实际工作流。
4. 对共享 Skill 生成契约，生成或模拟至少一个完整目标候选，并验证目标事实来自证据而非生成器。
   当契约声称广泛可移植时，使用存在实质差异的目标。
5. 结构化比较语言镜像，并保留命令、相对路径、标识符、代码块、分类和行为含义。
6. 运行仓库格式和 diff 完整性检查，然后再次检查完整最终 Skill，而不是只审查修改行。
7. 仍有占位符、绝对路径字面量、不受支持的声明、陈旧资源、失败检查或未解决所有权冲突时，不得报告成功。

## 结果

报告产物属于项目私有 Skill、共享 Skill 还是共享 Skill 生成契约，并报告所选契约结构、保留方向、
从全局重写的职责、删除的重复或陈旧内容、拥有的资源、同步 surface 和精确验证结果。
