# 编写 Skill

写出一份其他 Agent 不需要额外背景就能使用的 Skill：系统能正确发现，内容能照着执行，而且每项做法
都有证据支撑。每次调用都要按当前需求重建这份 Skill 适用范围内的整套工作，不要沿用旧改动留下的结构。

## 分类

动笔前，先根据 Skill 的分发方式、职责归属和用户要的产物判断类型。

| 判断依据 | 类型 |
| --- | --- |
| 整项工作由一个仓库维护并执行 | 项目本地 Skill |
| 发布出去的 Skill 会在不同仓库中直接执行同一套稳定流程 | 共享 Skill |
| 发布出去的内容负责写出一份由目标项目自己维护的完整 Skill | 共享 Skill 生成契约 |

删掉本地细节，并不会自动变成共享 Skill。只有工作流本身在不同仓库中都能保持稳定，而且能在运行时
发现目标项目的具体事实，才把它归为共享 Skill；否则继续作为项目本地 Skill。

项目本地 Skill 和共享 Skill 都可以负责操作、诊断或编排。只有这种区分会改变谁负责、怎么执行、
要通过哪些审查关口或什么条件下才算完成时，才需要细分。

## 证据

只查那些可能影响 Skill 行为、归属、资源或验证方式的事实。

### 任务和失败预期

弄清用户真正想要的结果、触发场景、限制、示例、兼容要求，以及失败时应该怎样处理。写明工作从什么
状态开始、做到什么程度才算完成、遇到什么情况必须停下，以及无法完成时要报告什么。

现有 Skill 只用来核对事实和防止遗漏，不要把它当成必须沿用的提纲。可复用的执行方法要和变更记录、
项目规则、个人偏好，以及本应放在项目文档里的说明分开。

### 负责范围的证据

检查这个 Skill 管理的整个目录，包括脚本、参考资料、素材、语言镜像、测试和实际用法，还要检查所有
可能随它变化的发现入口和分发入口。如果是项目本地 Skill，还要核对当前代码、配置、命令、服务、生成
内容由谁维护，以及项目实际怎样验证。凡是写进 Skill 的具体说法，都要先在当前仓库里证实。

### 跨环境与生成证据

如果是共享 Skill，要查看有代表性的仓库和平台，找出在哪些环境里都能直接执行的最小行为，并写清
哪些目标信息必须在运行时再发现。如果是共享 Skill 生成契约，要先确定所有目标项目都必须收集哪些
证据，再检查至少一个有代表性的目标，以及候选 Skill 的生成、审查、验收和交接分别由哪里负责。

## 编写

1. 先说清楚这项工作做什么、何时触发、要得到什么结果、从哪里开始、何时完成、什么情况必须停下、
   失败后怎么办，以及哪些事情不归它管。
2. 按当前用户意图和证据重新整理出一份完整版本。保留用户想要的结果、已经核实的决定和安全边界，
   删掉过时、重复、冲突或放错位置的内容与资源。假设今天从零开始写这个 Skill，按最合理的顺序组织
   它；每项要求都放到应在的位置，不要为了缩小 diff 而在末尾补一句，也不要机械保留旧顺序。
3. 每项要求都写进负责它的 Skill 章节或 Skill 自有资源。用户点名的 Skill 不负责某项要求时，不要写
   进去。只有负责该要求的 Skill 已经在用户请求范围内时，才能直接修改；否则先取得用户明确同意。
4. 只修改 Skill，以及执行或分发这些 Skill 所必需的自有资源、平台适配文件、索引、清单、语言镜像和
   契约测试。其他类型的改动都超出范围，必须报告。
5. 不看 diff，从头读一遍完整成稿。不断精简和调整，直到每一段都提供不重复、仍然有效而且能照着
   执行的信息。

每条指令都只能有一种明确解读。执行者、触发条件、适用条件、动作、停止或完成条件和预期结果等信息，
只要会影响执行，就必须写明。除非有新证据或用户明确同意，不得扩大、缩小、弱化、强化或改写原意。

重点写清要达到的结果、相关事实和判断边界，不要把篇幅浪费在过程复述上。如果有多种做法都合理，
就给执行 Agent 留出选择空间；如果顺序会影响正确性或安全性，就明确规定步骤。如果某项操作重复、固定
或容易出错，优先交给脚本，而不是每次临时生成一套说明。

纳入版本控制的 Skill 只能写仓库相对路径、Skill 根目录相对路径，或协议规定的稳定路径，不能写某台
机器专用的绝对路径。运行时工具确实需要绝对路径时，应从已经找到的根目录或任务输入中计算出来，
不要把它固化进 Skill。

提到其他 Rule 或 Skill 时，使用目标系统声明或认可的正式名称，不要写它在文件系统里的路径。
只有文件或资源本身归当前契约管理，而且位置就是契约的一部分时，才使用路径。

根据 Skill 类型确定执行方式和内容边界：

- 项目本地 Skill 要写入已经核实的项目事实，并以完成当前仓库要求的结果为终点；项目要求仍留在 Rule；
- 共享 Skill 要把项目特有的假设改成运行时发现、Skill 自己管理的资源、协议规定的稳定路径和明确的
  停止条件，不同目标环境仍要得到同一项受支持的结果；
- 共享 Skill 生成契约要把编写、成品契约、审查、验收和交接分开，并且不能把目标 Skill 将来的实际
  工作流塞进生成说明里。

## Skill 契约

### 正文

开头先写用于发现 Skill 的元数据：

```markdown
---
name: lowercase-hyphenated-name
description: Use when [concrete triggers and situations].
---
```

frontmatter 只能包含 `name` 和 `description`。`name` 最多 64 个字符，并且必须和目录名一致。
`description` 要让系统能可靠判断何时启用这个 Skill，但不要提前概括启用后才会加载的工作流。

元数据后先写一个一级标题，再用一小段话说明目标和边界。正文要让人能找到：谁负责、从哪里开始、
何时完成、何时停止、失败怎么处理、如何验证、最后如何交接。只有这项工作确实需要时才增加章节，
不要为了套通用模板而凑章节。

正文剩余部分按实际职责和适用条件选择：

- 项目本地的操作或诊断 Skill 要包含证据或前置条件、工作流程或阶段、停止条件、验证和结果；
- 项目本地编排 Skill 要写清 Ownership、Managed Assets、Workflow、审查或验收关口、Validation 和
  Output，但不能把各组件自己的契约收进来；
- 共享的操作或诊断 Skill 可以使用直接执行的结构，但修改前必须先发现目标项目的真实情况；
- 共享编排 Skill 必须包含 Ownership、Managed Assets、Reconciliation Workflow、独立的 Review Gate
  和 Acceptance Gate、Validation 以及 Output，目标项目特有的要求仍由项目本地 Rule 负责；
- 共享 Skill 生成契约必须包含 Evidence、Authoring Workflow、Generated Skill Contract、Review Gate、
  Acceptance Gate 和 Handoff。

### 资源

关键决定留在 `SKILL.md`。只有正文明确说明何时读取时，才把条件性内容或大量细节拆到直接引用的资源
里，而且引用最多深入一层。只有重复、固定或脆弱的操作才值得增加脚本；脚本必须写清依赖、报错、恢复
方法，并有安全且有代表性的测试。只有 Skill 确实会用某项素材生成产物时，才添加该素材。

除非外部打包规范明确要求，否则不要添加 README、变更日志、安装说明或速查文件。每项要求只保留一个
权威来源；平台适配文件只保留平台必需的元数据，再加一个指向源文件的引用。

### 审查、验收与交接

需要同时审查和验收时，必须把它们作为两个独立结论。对于共享 Skill 生成契约：

- 先由 Review Gate 审查完整候选 Skill；
- 再由 Acceptance Gate 把完整成品放进有代表性的目标项目环境，实际执行它的工作流并完成验收；
- 两道关口都通过后才能进入 Handoff。

审查和验收都通过后才能交接。交接内容至少包括通过验收的候选 Skill、审查结论、验收证据，以及未运行
或未解决的事项。任何一道关口没有通过，都要停止交接并报告，不能把候选 Skill 当成已验收成品交出去。

### 脚本入口

项目本地 Skill 可以选择与项目已有语言和运行时匹配的脚本，例如 Python 项目使用 Python、Dart 项目
使用 Dart；也可以同时提供 `.sh` 和 `.ps1` 入口。

任何共享 Skill，包括共享 Skill 生成契约，自己管理的每套脚本流程都必须同时提供 `.sh` 和 `.ps1`
入口。两者必须达到同一项受支持的结果，但可以保留有证据依据的平台差异。

生成契约产出的项目本地 Skill 仍按上面的本地规则选择脚本；这些脚本不会因为生成器是共享的而自动
变成共享脚本。

## 验证

### 成品检查

- 从头阅读最终 Skill，不要只看改动行；确认分类、归属、发现元数据、目标、开始与完成条件、停止条件、
  失败处理、验证方式、资源和交付都清楚且一致。
- 确认改动只限于 Skill 及其管理的文件和入口。
- 检查最终成稿是否存在歧义或语义偏移：每条指令必须只有一种明确解读；只有出现新证据或用户明确同意时，
  才能改变原意。任一条件不满足都不能通过。
- 按职责逐个检查子节。适用于所有类型的要求要留在公共内容里；有条件的要求要放在真正负责它的主题
  下并写明适用条件；不要只为了结构对称而给每种类型分别开子节。
- 对照语言镜像的结构，确保路径、命令、标识符、代码块、分类和行为没有走样。

### 实际执行

- 选择项目匹配方案时，使用项目已有的语言和运行时，以安全、有代表性的输入测试脚本；
- 选择成对入口时，只验证当前平台对应的入口：Windows 运行 `.ps1`，其他平台运行 `.sh`；不要求在
  当前宿主验证另一个入口；
- 测试这次改动涉及的工作流和资源，并覆盖正常完成、相关停止条件、失败处理、明确报错和恢复方式；
- 共享 Skill 要选择有代表性的环境测试，覆盖运行时发现、项目规则优先级和停止条件；如果声称能广泛
  移植，还要选择差异明显的多个环境；
- 共享 Skill 生成契约要先审查完整候选 Skill，再把至少一份完整的目标 Skill 放进有代表性的目标项目
  环境中实际执行。生成出来的 Skill 按自身类型验证脚本。

### 分发检查

对每项自有资源和每个必需的发现入口运行当前项目规定的验证器、契约测试和 diff 完整性检查。确认适用的
平台适配文件、索引、清单、语言镜像和其他分发入口仍然一致。

只要证据还有疑点、行为没有事实支持、自有资源或必需的发现入口已经过时或无法访问，或者必需检查失败
或没有报告，就不能声称工作成功。

## 结果

交付时说明：产物属于哪一类、由哪个产物或仓库负责、最终文档结构和关口、保留了哪些决定、删除或
迁移了哪些内容与资源、更新了哪些发现入口、分发入口和语言镜像，以及各项验证的具体结果。
