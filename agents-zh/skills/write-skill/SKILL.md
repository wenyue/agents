# 编写技能

写出一份无需额外背景也能直接执行的完整技能契约。动笔前先判断它如何分发、承担什么职责；
无论用户只指出一个问题还是要求整体重写，都要对整份技能的质量负责。

## 整体重写原则

- 每次创建或更新技能，都要从它现在应当完成的完整任务出发，重新组织全文。
- 现有技能及其资源只是证据和查漏清单，不是必须沿用的框架或大纲。
- 在决定保留哪些旧表述前，先重新确认触发条件、结果、分发方式、所有权、边界、流程、失败处理、
  验证方式和配套资源。
- 先形成完整的新版本，再整体替换归自己负责的文件。差异只是交付手段，不要为了缩小 diff
  而牺牲技能质量。
- 把每项有效要求放到真正负责它的位置，删除已经失效、重复、矛盾、放错位置或只是在记录修改经过的内容。
- 不要为了回应这次请求，在末尾补一段例外、说明、附录或补丁式章节。应直接改写真正约束该问题的章节，
  然后重新审阅全文。
- 只要当前意图和证据仍然支持，就保留技能既定方向；但不要因为某种结构或说法已经存在，
  就把薄弱结构和过时表述一起留下。

## 写之前先分类

先判断文件归谁所有。共享文件要进一步区分：它是直接执行流程的共享技能，还是用来编写目标项目技能的
共享技能生成契约。判断时以清单、目录、同步归属、安装范围和用户要求的产物为证据。

| 类型 | 用途和判断依据 | 可以写到多具体 | 必须守住的边界 |
| --- | --- | --- | --- |
| 项目本地技能 | 服务于一个目标仓库，由该仓库负责；手写或生成都一样 | 可以写当前仓库里的相对路径、真实命令、服务、模块和项目规则 | 以目标项目的准确性为先；不要发布或泛化未经证实的事实 |
| 共享技能 | 分发到多个仓库，直接执行工作流程 | 只写跨仓库都成立的不变量、技能自带资源的相对路径，以及共享协议统一规定的稳定路径 | 运行时再解析项目事实，具体政策交给目标项目规则 |
| 共享技能生成契约 | 分发到多个仓库，用目标项目证据编写一份完整的目标技能 | 只写语义层面的证据类别，以及共享协议统一规定的稳定路径 | 描述如何编写和目标技能应具备什么；不要冒充最终技能 |

项目本地技能通常直接写明该仓库最终执行的流程。只有明确需要“项目本地技能生成契约”时，
才采用生成契约的写法，而且证据和产物仍归该仓库所有。不能只删掉具体路径，就把目标项目专用技能
包装成共享技能；要么围绕真正跨仓库稳定的契约重新设计，要么继续让它留在本地。

## 证据

完整阅读归该技能负责的整个目录，并按文件类型收集证据。

通用证据包括：

- 用户要求的结果、例子、限制和兼容性要求；
- 现有 `SKILL.md`、脚本、参考资料、资源文件、语言镜像、测试和真实使用文件；
- 适用的所有权规则、清单、验证器、包装文件、目录和权威规范。

不同类型还要补充以下证据：

- 项目本地技能：检查当前项目规则、代码、配置、命令、生成文件归属、服务和验证方式。
  项目事实优先于通用最佳实践。
- 共享技能：找出在所有支持的仓库和平台中都成立的最小行为。单个仓库的事实只能作为例子，不能当成默认值。
- 共享技能生成契约：检查目标技能需要读取哪些目标证据，并用差异足够大的代表性仓库检验这份契约。

把稳定的执行要求，与实现经过、旧改动、个人偏好以及本应放进项目文档的解释分开。

## 路径与所有权

- 跟踪进仓库的技能或资源不得写死绝对路径。不要写盘符、用户目录、检出目录或机器专用的临时目录。
- 工具在运行时确实需要绝对路径时，应从技能根目录、仓库根目录、任务输入或其他已发现的责任方动态解析，
  不能把解析后的机器路径写进技能文件。
- 技能自带的脚本、参考资料和资源文件都使用相对技能根目录的路径。参考资料只放一层，
  并明确说明什么时候需要读取。
- 项目本地技能使用相对仓库根目录的项目路径；只有在当前项目证据确实证明后，才能写入具体命令。
- 共享技能应使用语义化的项目目标并在运行时发现具体位置。只有共享协议在所有支持目标上都明确规定了某条路径，
  才能保留具体路径。
- 共享技能生成契约应从语义上描述目标文件；只有共享生成协议本身拥有的路径才可写死。
  生成出的项目技能必须根据目标证据解析自己的具体路径。
- 所有技能都适用的编写约束放在 `write-skill`；目标项目事实和行为政策放进对应项目的规则，
  技能引用它，不要复制一份。
- 每条指令只保留一个唯一来源。除非平台确实要求额外运行字段，包装文件只能包含平台元数据或源文件引用。

## 必备格式与契约结构

每份技能都要包含：

1. 只含 `name` 和 `description` 的 YAML frontmatter，除非目标项目另有约定。
2. 与目录名一致、全小写、用连字符分隔且不超过 64 个字符的名称。
3. 清楚说明技能做什么、在什么具体场景下触发的描述。所有触发信息都要写在这里，因为只有技能触发后才会加载正文。
4. 一个一级标题，随后用一小段话说明预期结果和责任边界。
5. 命令式要求、明确所有权、有证据支持的决定、验证方式，以及定义清楚的结果或交付物。

分类后再选择契约结构。某一职责确实需要时才增加章节，不要把多个完整模板拼在一起。

| 契约结构 | 章节顺序 | 额外要求 |
| --- | --- | --- |
| 项目本地操作或诊断技能 | Preconditions 或 Evidence；Workflow 或 Phases；Stop Conditions；Validation；Result | 使用已验证的目标项目事实，并以完成目标任务为终点 |
| 项目本地编排技能 | Ownership；Managed Assets；Workflow；如果需要把草案交给人审查，再设置 Review 或 Acceptance 门槛；Validation；Output | 只协调目标项目的各个组成部分，不接管它们各自的实现契约 |
| 共享操作或诊断技能 | Preconditions 或 Evidence；Workflow 或 Phases；Stop Conditions；Validation；Result | 只保留跨仓库行为，修改前先解析本地事实 |
| 共享编排技能 | Ownership；Managed Assets；Reconciliation Workflow；Review Gate；Acceptance Gate；Validation；Output | 编排、审查和接受必须分开，不能与生成或委派的实现混为一谈 |
| 共享技能生成契约 | Evidence；Authoring Workflow；Generated Skill Contract；Review and Handoff | 生成完整的目标项目技能草案；不要把编写说明和生成后的运行流程混在一起 |

## 工作流程

1. 判断文件属于项目本地技能、共享技能还是共享技能生成契约。起草前先记下支持这一判断的证据。
2. 明确这份技能应完整完成什么、明确不负责什么。能从证据确定的细节直接确定；
   只有未解决的歧义会实质改变行为时才向用户提问。
3. 盘点现有每项声明和资源，逐项标记为保留、重写、删除，或迁移到真正的责任方。
4. 先从头确认技能的不变量、完成条件、停止条件和跨技能边界，再决定是否复用旧标题或句子。
5. 选择契约结构，按规定顺序列出完整草案。
6. 写好 frontmatter，准确限定触发场景。除非目标运行时明确要求并支持，否则删除非标准元数据。
7. 把正文写成一份从触发到交付都完整连贯的契约。保留的事实应放回自然归属的章节，
   不要为了照顾旧文而沿用原顺序。
8. 按渐进披露设计资源。核心决定留在 `SKILL.md`；只有条件性内容或细节会妨碍主流程阅读时，才拆到直接引用的资源中。
9. 同步更新所有归自己负责的资源、语言镜像、公开清单、包装文件和契约测试。
   保留无关的项目文件和现有 Git 状态。
10. 不看 diff，直接通读完整草案，再逐项执行防退化检查和验证。任何一项不通过就继续重写。

## 内容与资源

- 只写不显而易见的流程知识、领域限制和决策；内容应恰好足够让另一位有能力的代理可靠完成任务。
- 一项技能只承担一个完整、内聚的工作单元。无关职责拆开，但不要把必须一起加载的一条端到端流程拆成多个技能。
- 具体程度要与风险匹配：上下文容易变化时说明目标和原因；修改风险高时给出准确顺序；
  必须保证确定性时使用可执行脚本。
- `SKILL.md` 应精简到每条指令都值得代理注意。核心流程难以浏览前，就把条件性细节移到参考资料；
  除非目标运行时明确允许其他限制，主文件应保持在 500 行以内。
- 只有重复出现、必须确定执行或容易手工出错的操作才值得加脚本。脚本要在安全、有代表性的输入上测试，
  并明确依赖、错误和恢复方式。
- 只有正文明确说明何时读取时才添加参考资料；只有技能会直接用它产出结果时才添加资源文件。
- 只有例子能消除歧义时才使用。不要解释代理本来就懂的概念、复述显而易见的步骤，
  或描述某条指令是如何被修改出来的。
- 除非外部打包标准明确要求，否则不要在技能目录里创建 README、changelog、安装说明或速查表。

## 防退化检查

出现以下任一情况，都要推翻并重写当前草案：

- 最新要求只是以附加说明出现，没有改到真正管这件事的契约；
- 同一项义务在多个章节重复，或与旧例外冲突；
- 项目本地事实混进共享契约，或为了所谓通用性把本地技能削弱成空泛建议；
- 共享技能生成契约写入最终目标流程，或共享技能反过来教人如何编写另一项技能；
- 仍有过时名称、路径、假设、占位符或已废弃流程阶段；
- 文档变长了，却没有增加独立职责或必要证据；
- 只是为了减小 diff 而保留旧章节顺序；
- 正文在讲修改历史、审查讨论或最近修了什么；
- 资源重复正文，或正文没有明确指向该资源。

验收前，以下问题必须全部回答“是”：

- 如果今天从零开始写，这会是最终写出的技能吗？
- 它的项目本地或共享归属是否明确，并得到目录或仓库证据支持？
- 如果是共享文件，能否明确看出它是共享技能还是共享技能生成契约？
- 每段话是否都只承担一项独立、现行、可执行的职责？
- 代理能否看清开始条件、完成条件、停止条件和交付要求？
- 除非职责确实扩展，整体复杂度是否没有增加，最好还有下降？

## 验证与交付

1. 运行当前技能验证器，以及覆盖清单、包装文件、语言镜像、同步和所有权的项目契约测试。
2. 对项目本地技能，要在目标仓库的真实配置和验证入口中执行已改脚本和有代表性的工作流程。
3. 对共享技能，要在有代表性的场景中试用真实流程，覆盖运行时发现、目标规则优先级和停止条件。
4. 对共享技能生成契约，至少生成或模拟一份完整的目标草案，确认目标事实来自证据而不是生成器臆测。
   如果契约声称适用范围很广，还要选择差异足够大的多个目标验证。
5. 对语言镜像做结构比对，确认命令、相对路径、标识符、代码块、分类和行为含义都未改变。
6. 运行仓库格式和差异完整性检查。最后再通读一次完整技能，不能只看改动行。
7. 只要还有占位符、绝对路径字面量、无证据声明、过时资源、失败检查或未解决的所有权冲突，
   就不能宣告完成。

## 交付结果

说明它属于项目本地技能、共享技能还是共享技能生成契约；同时报告采用的契约结构、保留的既定方向、
整体重写的职责、删掉的重复或过时内容、归自己负责的资源、已同步的界面，以及每项验证的准确结果。
