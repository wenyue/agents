# 编写 Skill

创建一份无需隐藏上下文即可由其他 Agent 使用、可发现、可执行且具有充分证据的 Skill。每次调用都重新
构建完整的责任工作，不保留以往编辑形成的结构。

## 分类

编写前根据分发和所有权证据选择一种类型。

| 类型 | 负责内容 | 具体程度 |
| --- | --- | --- |
| 项目本地 Skill | 一个仓库中的完整工作 | 已验证的仓库相对路径、命令、服务、模块、规则和停止条件 |
| 共享 Skill | 跨仓库直接执行的稳定行为 | 跨仓库不变量、Skill 自有资源、运行时发现和协议拥有的路径 |
| 共享 Skill 生成契约 | 编写完整的目标项目自有 Skill | 目标证据类别和生成 Skill 的契约，而非最终项目流程 |

目录、清单、安装范围、同步所有权和请求的输出比泛化措辞更能证明分类。删除本地细节不会让 Skill
自动成为共享 Skill；应围绕稳定行为和运行时发现重新设计，否则保留为项目本地 Skill。

直接执行的 Skill 可以是操作、诊断或编排契约。只有这种区分会改变所有权、执行方式、审查关卡或
完成条件时才使用它。

## 证据

只收集会改变 Skill 行为、所有权、资源或验证方式的证据：

- 用户的目标、触发条件、约束、示例、兼容性要求和失败预期；
- 完整的 Skill 责任目录、脚本、Reference、Asset、镜像、测试和真实用法；
- 适用的项目规则、清单、目录、Wrapper、验证器和权威规范；
- 对项目本地 Skill，检查当前代码、配置、命令、服务、生成内容所有权和验证行为；
- 对共享文件，找出在所支持仓库和平台中始终成立的最小行为或证据类别。

把现有 Skill 当作证据和防遗漏检查，而不是首选提纲。区分可复用执行知识与变更历史、项目政策、
个人偏好，以及应放在项目文档中的解释。

## 编写

1. 明确完整工作、触发条件、目标、开始条件、完成条件、停止条件、失败行为和 Skill 排除的职责。
2. 从当前意图综合完整候选文件。保留仍有依据的方向和安全边界；删除过时、重复、矛盾或位置错误的
   内容和资源。
   按照今天从零编写该 Skill 时应有的结构组织结果，将每项要求整合进其责任阶段，不为缩小 diff 而
   追加说明或保留旧顺序。
3. 将每项要求放到自然所有者中。项目政策保留在规则中；Skill 只引用，不复制。
4. 在执行方式或分发需要时，同步更新所属资源、Wrapper、清单、镜像和契约测试。
5. 脱离 diff 阅读完整候选文件，持续修改，直到每段内容都提供独立、现行、可执行的信息。

优先描述目标结果、相关事实和决策边界，不叙述不必要的流程。上下文决定最佳做法时给执行 Agent 留出
自由；顺序影响正确性或安全性时规定步骤；重复、确定或脆弱的行为由脚本执行比临时生成说明更可靠。

受版本控制的 Skill 只使用仓库相对路径、Skill 根目录相对路径或协议拥有的稳定路径，不使用机器特有的
绝对路径。运行时工具需要绝对路径时，从已发现的根目录或任务输入推导，不把它写入 Skill。

引用其他 Rule 或 Skill 时，使用目标系统声明或识别的规范名称，绝不使用其文件系统路径。只有当前契约
所拥有且位置属于契约的文件或资源才使用路径。

## Skill 契约

以发现元数据开头：

```markdown
---
name: lowercase-hyphenated-name
description: Use when [concrete triggers and situations].
---
```

在本仓库约定下，frontmatter 只包含 `name` 和 `description`；名称不超过 64 个字符且与目录一致。
description 应支持可靠选择，但不概括激活后才加载的工作流。

元数据后写一个一级标题和一段简短的目标与边界说明。在正文中清楚呈现所有权、开始、完成、停止、失败、
验证和交付。只有工作本身需要时才增加章节，不要为了满足通用模板而增加章节。

| 契约结构 | 必需内容 | 额外要求 |
| --- | --- | --- |
| 项目本地操作或诊断 | Evidence 或 Preconditions；Workflow 或 Phases；Stop Conditions；Validation；Result | 使用已验证的项目事实并完成请求目标 |
| 项目本地编排 | Ownership；Managed Assets；Workflow；Review 或 Acceptance gates；Validation；Output | 协调组件但不吸收其契约 |
| 共享操作或诊断 | Evidence 或 Preconditions；Workflow 或 Phases；Stop Conditions；Validation；Result | 修改前发现目标事实 |
| 共享编排 | Ownership；Managed Assets；Reconciliation Workflow；Review Gate；Acceptance Gate；Validation；Output | 分开编排、审查和验收 |
| 共享 Skill 生成契约 | Evidence；Authoring Workflow；Generated Skill Contract；Review and Handoff | 生成完整的目标项目自有候选文件，不混合编写说明和运行时流程 |

核心决定保留在 `SKILL.md`。只有正文写明读取条件时，才把条件性或详细内容移入直接引用的资源；引用深度
保持一层。只为重复、确定或脆弱的操作增加脚本，并明确依赖、错误、恢复方式和安全代表性测试。只有 Skill
直接用 Asset 生成输出时才增加 Asset。

外部打包契约没有要求时，不增加 README、Changelog、安装或快速参考文件。每项要求只保留一个事实源；
Wrapper 只包含平台必需元数据和一个源引用。

## 验证

- 阅读最终 Skill 而不只看变更行，验证分类、所有权、发现元数据、目标、开始、完成、停止、失败、验证、
  资源和交付。
- 对项目本地 Skill，根据当前仓库事实运行发生变化的工作流和资源。
- 对共享 Skill，在覆盖运行时发现、项目规则优先级和停止条件的代表性环境中测试。
- 对生成契约，至少模拟一份完整目标 Skill；声称广泛可移植时使用差异足够大的多个目标。
- 在安全代表性输入上测试发生变化的脚本，并确认明确的错误和恢复行为。
- 按结构比较语言镜像，保留路径、命令、标识符、代码块、分类和行为。
- 对所有所属资源和发现入口运行当前验证器、契约测试和 diff 完整性检查。

证据仍未解决、资源过时或不可达、行为缺乏支持，或必需检查失败或未报告时，不得声称成功。

## 结果

报告文件类型、自然所有者、最终契约结构、保留的决定、删除或迁移的内容与资源、更新的发现入口和镜像，
以及准确的验证结果。
