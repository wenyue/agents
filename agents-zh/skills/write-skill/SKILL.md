# 编写 Skill

写出一份其他 Agent 不需要额外背景就能使用的 Skill：系统能正确发现，内容能照着执行，而且每项做法
都有证据支撑。每次调用都要按当前需求重建它负责的整套工作，不要沿用旧改动留下的结构。

## 分类

动笔前，先根据分发方式和职责归属判断它属于哪一类。

| 类型 | 负责什么 | 需要具体到什么程度 |
| --- | --- | --- |
| 项目本地 Skill | 为一个仓库完成一整项工作 | 写明已经核实的仓库相对路径、命令、服务、模块、规则和停止条件 |
| 共享 Skill | 在不同仓库里直接执行同一种稳定行为 | 写清跨仓库都成立的约束、Skill 自己管理的资源、运行时发现方式和协议规定的路径 |
| 共享 Skill 生成契约 | 指导生成一份由目标项目自己维护的完整 Skill | 规定目标项目要收集哪些证据，以及最终 Skill 必须满足什么契约；不要把目标项目的实际流程写死 |

资产目录、清单、安装范围、同步归属和用户要的产物，比表面上是否“通用”更能说明它属于哪一类。
删掉本地细节，并不会自动变成共享 Skill。要么围绕真正稳定的行为和运行时发现重新设计，要么继续
作为项目本地 Skill。

直接使用的 Skill 可以负责操作、诊断，也可以充当编排契约。只有这种区分会改变谁负责、怎么执行、
要通过哪些审查关口或什么条件下才算完成时，才需要细分。

## 证据

只查那些可能影响 Skill 行为、归属、资源或验证方式的事实：

- 用户真正想要的结果、触发场景、限制、示例、兼容要求，以及失败时应该怎样处理；
- 这个 Skill 管理的整个目录，包括脚本、参考资料、素材、语言镜像、测试和实际用法；
- 适用的项目规则、清单、资产目录、平台适配文件（wrapper）、验证器和权威规范；
- 如果是项目本地 Skill，核对当前代码、配置、命令、服务、生成内容由谁维护，以及项目如何验证；
- 如果是共享内容，只保留在所有支持的仓库和平台上都成立的最小行为与证据要求。

现有 Skill 只用来核对事实和防止遗漏，不要把它当成必须沿用的提纲。可复用的执行方法要和变更记录、
项目规则、个人偏好，以及本应放在项目文档里的说明分开。

## 编写

1. 先说清楚这项工作做什么、何时触发、要得到什么结果、从哪里开始、何时完成、什么情况必须停下、
   失败后怎么办，以及哪些事情不归它管。
2. 按当前需求重新整理出一份完整版本。保留仍有依据的方向和安全边界，删掉过时、重复、冲突或放错
   位置的内容与资源。假设今天从零开始写这个 Skill，按最合理的顺序组织它；每项要求都放进真正负责
   它的阶段，不要为了缩小 diff 而在末尾补一句，也不要机械保留旧顺序。
3. 每项要求都交给本该负责它的规则、Skill 或资源。项目要求留在规则里，Skill 只引用，不要复制一份。
4. 如果执行或分发方式需要，同步更新它管理的资源、平台适配文件、清单、语言镜像和契约测试。
5. 不看 diff，从头读一遍完整成稿。不断精简和调整，直到每一段都提供不重复、仍然有效而且能照着
   执行的信息。

重点写清要达到的结果、相关事实和判断边界，不要把篇幅浪费在过程复述上。如果具体做法要看上下文，
就给执行 Agent 留出选择空间；如果顺序会影响正确性或安全性，就明确规定步骤；如果某项操作重复、固定
或容易出错，优先交给脚本，而不是每次临时生成一套说明。

纳入版本控制的 Skill 只能写仓库相对路径、Skill 根目录相对路径，或协议规定的稳定路径，不能写某台
机器专用的绝对路径。运行时工具确实需要绝对路径时，应从已经找到的根目录或任务输入中计算出来，
不要把它固化进 Skill。

提到其他 Rule 或 Skill 时，使用目标系统声明或认可的正式名称，不要写它在文件系统里的路径。
只有文件或资源本身归当前契约管理，而且位置就是契约的一部分时，才使用路径。

## Skill 契约

开头先写用于发现 Skill 的元数据：

```markdown
---
name: lowercase-hyphenated-name
description: Use when [concrete triggers and situations].
---
```

按本仓库约定，frontmatter 只能包含 `name` 和 `description`。`name` 最多 64 个字符，并且必须和目录名
一致。`description` 要让系统能可靠判断何时启用这个 Skill，但不要提前概括启用后才会加载的工作流。

元数据后先写一个一级标题，再用一小段话说明目标和边界。正文要让人能找到：谁负责、从哪里开始、
何时完成、何时停止、失败怎么处理、如何验证、最后如何交接。只有这项工作确实需要时才增加章节，
不要为了套通用模板而凑章节。

| 契约形式 | 必须包含 | 额外要求 |
| --- | --- | --- |
| 项目本地操作或诊断 Skill | Evidence 或 Preconditions；Workflow 或 Phases；Stop Conditions；Validation；Result | 使用已经核实的项目事实，并以完成用户要求为终点 |
| 项目本地编排 Skill | Ownership；Managed Assets；Workflow；Review 或 Acceptance gates；Validation；Output | 负责协调各组件，但不要把组件自己的契约收进来 |
| 共享操作或诊断 Skill | Evidence 或 Preconditions；Workflow 或 Phases；Stop Conditions；Validation；Result | 修改任何内容前，先发现目标项目的实际情况 |
| 共享编排 Skill | Ownership；Managed Assets；Reconciliation Workflow；Review Gate；Acceptance Gate；Validation；Output | 把编排、审查和验收分成不同阶段 |
| 共享 Skill 生成契约 | Evidence；Authoring Workflow；Generated Skill Contract；Review and Handoff | 生成一份完整、由目标项目自己维护的候选 Skill；不要把编写说明和它将来运行的流程混在一起 |

关键决定留在 `SKILL.md`。只有正文明确说明何时读取时，才把条件性内容或大量细节拆到直接引用的资源
里，而且引用最多深入一层。只有重复、固定或脆弱的操作才值得增加脚本；脚本必须写清依赖、报错、恢复
方法，并有安全且有代表性的测试。只有 Skill 确实会用某项素材生成产物时，才添加该素材。

除非外部打包规范明确要求，否则不要添加 README、变更日志、安装说明或速查文件。每项要求只保留一个
权威来源；平台适配文件只保留平台必需的元数据，再加一个指向源文件的引用。

## 验证

- 从头阅读最终 Skill，不要只看改动行；确认分类、归属、发现元数据、目标、开始与完成条件、停止条件、
  失败处理、验证方式、资源和交付都清楚且一致。
- 项目本地 Skill：按照当前仓库的真实情况，实际运行这次改动涉及的流程和资源。
- 共享 Skill：选择有代表性的环境测试，覆盖运行时发现、项目规则优先级和停止条件。
- 生成契约：至少模拟生成一份完整的目标 Skill；如果声称能广泛移植，还要选择差异明显的多个目标。
- 脚本有改动时，用安全且有代表性的输入测试，并确认报错和恢复方式都明确有效。
- 对照语言镜像的结构，确保路径、命令、标识符、代码块、分类和行为没有走样。
- 对所有由它管理的资源和供系统发现它的入口，运行当前项目规定的验证器、契约测试和 diff 完整性
  检查。

只要证据还有疑点、资源已经过时或无法访问、行为没有事实支持，或者必需检查失败或没有报告，就不能
声称工作成功。

## 结果

交付时说明：产物属于哪一类、最合适的责任方是谁、最终采用哪种契约形式、保留了哪些决定、删除或
迁移了哪些内容与资源、更新了哪些系统发现入口和语言镜像，以及各项验证的具体结果。
