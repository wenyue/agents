# 变更集验证

根据目标仓库的当前证据，生成由目标仓库拥有的验证 skill。该 skill 应只处理一个内在一致且已经完成的变更集，以及可靠交付这组变更所必需的最少检查。

## 证据

阅读目标仓库的 agent 规则、工具配置、清单、测试布局、CI 工作流、生成文件的所有权、依赖边界，以及仓库自有的验证选择器。确定：

- 格式化、自动修复、静态分析、lint 和测试各自支持的范围；
- 测试的直接所有权，以及变更与检查之间的对应关系；
- 各工具是否会修改文件、运行前提、相对成本，以及安全的自动修复路径；
- 哪些风险要求扩大到子系统级或完整验证。

不得虚构证据无法证明的命令、所有权映射或范围支持。

## 编写流程

1. 对生成的 skill 必须处理的变更类型进行分类。
2. 为每种类型选择仓库支持的最小检查范围，并定义有证据依据的扩大范围条件。
3. 优先使用仓库已有的选择器。只有仓库自有工具无法可靠表达反复执行的确定性选择逻辑时，才添加 skill 自有脚本。
4. 生成 `SKILL.md`。只有多个软件包、语言或风险映射会妨碍核心流程的清晰表达时，才添加 `references/verification-matrix.md`。
5. 按照下述契约审查完整的生成目录。

## 生成 Skill 契约

### 触发时机与范围

- 在实现已经完成的检查点、交付之前运行。不得打断仍在进行的编辑、调试或尚未完成的修复循环。
- 根据任务上下文和仓库状态，确定生产代码、测试、配置、生成输出的源头及配套文件。
- 从满足需要的最小范围开始。只有依赖关系、共享契约、生成接口、自动修复产生的改动、工具限制或所有权不明使较小范围的结果不可靠时，才扩大范围。
- 测试所有权缺失是需要解决的缺口或扩大范围的理由，绝不能据此认定无需测试。
- 保留用户的无关工作。绝不编辑生成输出、第三方代码或所选项目自有范围之外的文件。

### 规范化与修复

1. 如果项目提供格式化工具，对选定的项目自有源代码范围执行格式化。
2. 每项获项目认可的自动修复，在其支持的最小选定范围内最多运行一次。如果同一工具同时负责静态分析和 lint，不要重复执行相同工作。
   接受该范围内各项诊断对应的机械性修复，无论这些诊断是否由当前变更引入。不得只为发现或修复既有违规而扩大范围。
3. 将格式化和自动修复改动的文件加入变更范围。如果工具改动了生成文件、第三方文件或其他不归项目所有的文件，立即停止。
4. 重新格式化被自动修复改动的源代码，再运行所支持的最小范围非修改型静态检查。
5. 将剩余的语义诊断连同准确位置和消息交回父实现 agent。不要在 verifier 中自行进行语义修复。
6. 父 agent 完成修改后，在下一个实现完成检查点重新开始规范化。

### 验证与结果

- 静态检查通过后，运行直接归属于所选范围的测试。格式化、自动修复或父 agent 的修改使组件被纳入范围时，也要运行这些组件的测试。
- 每个实现完成检查点内，各项不同的验证最多运行一次；只有发生修改，且有记录表明确需复查时，才再次运行。
- 将每个选定验证项归类为 `passed`、`failed`、`inconclusive` 或 `not applicable`。报告命令、范围、选择理由、结果和剩余缺口。
- 报告所有被改动的文件、格式化与自动修复调用、重复检查、剩余诊断和验证缺口。
- 返回一个总体结果：`passed`、`semantic_fix_required`、`failed` 或 `inconclusive`。
- 只要任何必需的验证项失败或仍无定论，就不得报告 `passed`。
- 如果范围外的失败可能早于当前变更，只将该失败项与可信基线比较。不得仅为分类而运行完整的基线测试套件。

## 可选资源

只有仓库证据证明有必要时，才包含可执行脚本。生成的 skill 如果包含脚本，就必须包含 `## Failure Recovery`：脚本失败时停止，报告准确的命令和错误，分析原因，并在修改或重试脚本前提出候选变更。

## 审查与交付

确认生成的 skill 与目标仓库的工具规则一致，不负责实现工作或 worktree 生命周期，也不包含未经证据支持的命令或映射。将完整生成目录及其支持证据交给 `setup-project-agents`，供其审查和验收候选方案。
