# 变更集验证

根据目标仓库的当前证据生成目标自有验证 skill。让它聚焦一个连贯的已完成变更集，以及可靠交付该变更所需的最小检查。

## 证据

读取目标仓库的 agent 规则、工具配置、manifest、测试布局、CI workflow、生成文件所有权、依赖边界和仓库自有验证选择器。确定：

- formatter、fixer、analyzer、linter 和测试支持的范围；
- 直接测试所有权和变更到检查的映射；
- 修改行为、前置条件、相对成本和安全的自动修复路径；
- 需要扩大到子系统或完整验证的风险。

不要发明证据无法证明的命令、所有权映射或范围支持。

## 编写工作流

1. 对生成 skill 必须处理的变更类型进行分类。
2. 为每种类型选择仓库支持的最窄检查，并定义有证据支持的扩大条件。
3. 优先使用现有仓库选择器。只有重复的确定性选择无法用仓库自有工具可靠表达时，才添加 skill 自有脚本。
4. 生成 `SKILL.md`。只有多 package、多语言或风险映射会使核心流程难以理解时，才添加 `references/verification-matrix.md`。
5. 按下述契约审查完整生成目录。

## 生成 Skill 契约

### 触发和范围

- 在实现完成检查点、交付之前运行。不要中断正在进行的编辑、调试或未完成的修复循环。
- 从任务上下文和仓库状态解析生产代码、测试、配置、生成输出所有者和支持文件。
- 从最小充分范围开始。只有依赖、共享契约、生成接口、fixer 修改、工具限制或未知所有权让较窄结果不可靠时才扩大。
- 将缺失的测试所有权视为要解决的缺口或扩大范围的理由，绝不能视为不需要测试的证明。
- 保留无关用户工作。绝不编辑生成输出、第三方代码或选定项目自有范围之外的文件。

### 规范化和修复

1. 当项目提供 formatter 时，格式化选定的项目自有源代码范围。
2. 每个项目批准的自动 fixer 在其支持的最小选定范围内最多运行一次。当一个工具同时负责 analyzer 和 linter surface 时，不要重复执行。
   接受该范围内发现的机械修复，无论当前变更是否引入了这些诊断。不要仅为发现或修复旧违规而扩大范围。
3. 将 formatter 和 fixer 修改的文件加入变更范围。如果工具修改生成文件、第三方文件或其他非自有文件，立即停止。
4. 重新格式化 fixer 修改的源代码，然后运行支持的最小非修改型静态检查。
5. 将剩余语义诊断连同准确位置和消息返回父实现 agent。不要在 verifier 内编写语义修复。
6. 父 agent 编辑后，在下一个完成检查点重新开始规范化。

### 验证和结果

- 静态检查通过后运行直接拥有的测试。包括 formatter、fixer 或父 agent 修改所扩展范围内组件的测试。
- 每个完成检查点只运行每个唯一验证 surface 一次，除非某次修改要求有记录的重复检查。
- 将每个选定 surface 分类为 `passed`、`failed`、`inconclusive` 或 `not applicable`。报告命令、范围、选择原因、结果和剩余缺口。
- 返回一个总体结果：`passed`、`semantic_fix_required`、`failed` 或 `inconclusive`。
- 任何必需 surface 失败或仍为 inconclusive 时，不要报告 `passed`。
- 当范围外失败可能早于当前变更时，只将该失败 surface 与可信 baseline 比较。不要仅为分类而运行完整 baseline suite。

## 可选资源

只有仓库证据证明需要时才包含可执行脚本。包含脚本的生成 skill 必须包含 `## Failure Recovery`：脚本失败时停止，报告准确命令和错误，分析原因，并在修改或重试脚本前提出候选变更。

## 审查和交付

确认生成 skill 与目标工具规则一致，不拥有实现或 worktree 生命周期行为，也不包含不受支持的命令或映射。将完整生成目录及其支持证据交给 `setup-project-agents` 进行候选审查和验收。
