# 变更集验证

生成一个归目标仓库所有的 skill，在交接前规范化并验证一个内聚且已完成的变更集。应根据当前仓库证据推导，而不是套用通用命令清单。

## 编写工作流

1. 阅读 `.agents/rules/20-project-tools.md`、包清单、测试布局、生成文件所有权、CI 配置、仓库自有选择器和依赖边界证据。
2. 识别变更类别、格式化器和静态分析支持的最小范围、直接归属测试、项目认可的自动修复器、特殊验证面，以及需要扩大或完整验证的风险。
3. 生成 `SKILL.md`。仅当仓库包含多语言或多包，或者“变更到检查”的映射会遮蔽核心工作流时，才添加 `references/verification-matrix.md`。
4. 优先使用仓库已有的选择器。只有当前证据证明确定性的重复逻辑无法由已有工具可靠表达时，才生成 skill 自有脚本。
5. 使命令、路径、范围支持、修改行为、前置条件和成本特征与 `.agents/rules/20-project-tools.md` 保持一致。

## 生成契约

- 当一个内聚且已完成的变更集准备交接时触发。跳过正在进行的实现、调试和迭代编辑；后续再次编辑后，在下一个完成检查点重新验证。
- 从任务上下文和仓库状态解析变更集涉及的生产代码、测试、配置、生成输出所有者及辅助文件。依据项目证据选择最小充分范围，不要默认执行全项目检查。
- 使用直接测试所有权、依赖关系、项目映射和可用的仓库选择器。每个完成检查点对每个独立验证面只运行一次。
- 从窄范围开始。仅当工具范围限制、依赖影响、测试基础设施、共享契约、生成接口、修复器修改或影响未知导致较小范围不可靠时，才扩大到子系统或完整验证。
- 不要把缺少测试映射视为无需测试的证据。行为发生变化且无法确定归属时，报告缺口或扩大范围。
- 保留用户无关改动，绝不手改生成输出或第三方代码。不要仅为清理选定范围之外的既有诊断而扩大范围。
- 将每个选定验证面分类为 `passed`、`failed`、`inconclusive` 或 `not applicable`。报告其命令、范围、选择原因、结果及剩余缺口。任何必要验证面失败或结论不确定时，不得宣称整体成功。
- 若选定范围之外的失败可能早于当前变更，只重新运行该失败验证面，并在需要时与可信基线上的同一验证面对比。绝不能仅为分类一个疑似既有失败而运行基线完整测试套件。

## 规范化与机械性修复

1. 若项目存在格式化器，在选定的项目自有源文件范围内运行。
2. 对每个项目认可的自动修复器，在其支持的最小选定范围内最多运行一次。分析器和 linter 的诊断可能共享一个工具面；不要重复检查或修复。
3. 接受选定范围内发现的所有机械性修复，无论是否由当前变更引入。不要仅为发现或修复旧问题而扩大范围。
4. 将格式化器或修复器修改的每个文件加入验证范围。工具若修改生成文件、第三方文件或其他非项目自有文件，立即停止并报告。
5. 重新格式化每个被修复器修改的源文件，然后运行工具支持的最小非修改性静态分析范围。
6. 将剩余语义诊断连同准确位置和消息返回父实现 agent。验证器不得自行编写语义修复。父 agent 编辑代码后，在下一个完成检查点从规范化开始重新执行本工作流。
7. 静态检查通过后运行直接归属的单元测试。为格式化器、修复器或父 agent 改动所加入范围的每个组件包含测试；累计影响使窄范围不可靠时扩大范围。
8. 报告每个修改文件、格式化器和修复器调用、重复检查、剩余诊断及验证缺口。

## 验证器结果

生成的 skill 必须返回一个整体结果：

- `passed`：所有必要的选定验证面均在机械性修复后通过。
- `semantic_fix_required`：静态分析发现需要父 agent 判断的诊断。
- `failed`：必要检查或测试失败，且无法有效归类为既有失败。
- `inconclusive`：前置条件、所有权歧义或证据不可用，导致无法得出可靠结论。

## 条件式脚本要求

仅当当前证据表明确定性的重复逻辑无法由仓库自有工具可靠表达时，才生成可执行脚本。若生成的 skill 包含可执行脚本，要求它包含自己的 `## Failure Recovery` 小节，指示 agent 立即停止、报告准确命令和错误、分析原因，并提出具体的候选修改。在提案得到审查前，不得继续验证、隐藏失败或重试已修改代码。

## 交接

把完整生成目录和辅助证据交给 `setup-project-agents`。候选项审查与验收由该工作流负责。
