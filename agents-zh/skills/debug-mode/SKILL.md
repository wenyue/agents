# 调试模式

你现在处于**调试模式**——一种假设驱动的调试工作流。不要直接跳到修复。按顺序完成每个阶段。

---

## 阶段 1：理解 Bug

如果尚未提供，询问用户：预期行为与实际行为、复现步骤、错误消息。

阅读相关源代码。在提出任何假设前，理解调用链和数据流。

## 阶段 2：提出假设

以编号列表形式提出**可检验的假设**：

```
根据分析，我的假设如下：

1. **[标题]** — [可能存在的问题及原因]
2. **[标题]** — [说明]
3. **[标题]** — [说明]
```

同时包含明显和不明显的原因：竞态条件、差一错误、陈旧闭包、null 或延迟初始化、类型强制转换、流或监听器泄漏，以及构建顺序问题。

## 阶段 3：插桩代码

### 日志文件

使用**绝对路径**写入 **`{project_root}/.agents/debug.log`**。

**`project_root` = 从上下文推导出的硬编码常量字符串**，例如对话中的文件路径。禁止使用：`Directory.current`、`Platform.script`、`__dirname`、`process.cwd()`、`path.resolve()` 或任何运行时检测。例外：远程或 CI 环境，或本地文件系统不可写时，改用 `/tmp/agents-debug.log`。

每次复现前：需要时创建 `.agents/`，然后**清空**日志。

- Dart/Flutter 和服务端：使用文件追加 API，例如 `File(path).writeAsStringSync(msg, mode: FileMode.append)`、`fs.appendFileSync` 或 `open("a")`。
- 浏览器或 Flutter Web：通过 `fetch`/`http.post` 调用把内容追加到文件的调试 API 路由。
- **必须在所有构建模式下工作**（debug/profile/release）。

### 区域标记

所有插桩都必须包在区域块中，以便干净移除：

```
// #region DEBUG       (Dart/JS/TS/Java/C#/Go/Rust/C/C++)
# #region DEBUG        (Python/Ruby/Shell/YAML)
<!-- #region DEBUG --> (HTML/Vue/Svelte)
-- #region DEBUG       (Lua/SQL)

...插桩代码...

// #endregion DEBUG    (与之匹配的结束标记)
```

### 日志规则

- **绝不使用 `print`、`debugPrint`、`console.log` 或任何 stdout/stderr 输出。**所有调试输出必须写入 `debug.log`：服务端通过文件追加，浏览器端通过 HTTP POST 到调试端点。
- 日志消息包含假设编号：`[DEBUG H1]`、`[DEBUG H2]` 等。
- 记录变量状态、执行路径、时序和决策点。
- 保持最小化——只记录确认或排除各假设所需的内容。

完成插桩后，要求用户复现 bug，然后**停止并等待**。

## 阶段 4：分析日志并诊断

用户完成复现后：

1. **先检查日志大小**（`wc -l` 或 `ls -lh`）。如果日志很大，使用 `tail` 或 `grep '\[DEBUG H'` 只提取相关行，不要读取整个文件。
2. 将日志映射到各假设，判断哪些已**确认**、哪些已**排除**。
3. 用证据给出诊断：

```
## 诊断

**根因**：[由日志证据支持的说明]

证据：
- [H1] 已排除 — [原因]
- [H2] 已确认 — [日志证据]
```

如果结论不确定：提出新假设 → 添加更多插桩 → 清空日志 → 再次要求用户复现。

## 阶段 5：生成修复

编写修复。**保留调试插桩。**

清空 `.agents/debug.log`，要求用户验证修复是否生效，然后**停止并等待**。

## 阶段 6：验证并清理

**如果已修复：**移除所有 `#region DEBUG` 块及其内容，删除 `.agents/debug.log`，并总结根因和修复。

**如果未修复：**读取新日志，询问用户观察到的情况，返回**阶段 2**并继续迭代。

---

## 规则

- **绝不跳过阶段。**即使自认为已经知道答案，也要插桩并验证。
- **用户确认修复前，绝不移除插桩。**
- **绝不使用 `print`/`debugPrint`/`console.log`。**所有调试输出只能通过文件追加写入 `.agents/debug.log`。
- **每次复现前始终清空日志。**
- **始终用 `#region DEBUG` 块包裹插桩。**
- 要求用户复现后，**始终等待用户**。

## 危险信号——停止

- “我已经看出 bug，直接修补就行。” → 仍然先插桩并验证。
- 即将调用 `print`/`debugPrint`/`console.log`。→ 改为向 `debug.log` 追加文件内容。
- 即将读取整个 `debug.log`。→ 改用 `grep '\[DEBUG H'` 或 `tail`。
- 用户确认修复前即将移除 `#region DEBUG`。→ 保留到阶段 6。
- 即将在运行时计算 `project_root`。→ 将其硬编码为常量字符串。
