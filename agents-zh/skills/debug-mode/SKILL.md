# Debug Mode

在修复 bug 前使用可检验假设、针对性文件日志仪表和人工复现进行诊断。按顺序执行各阶段，并在每个复现门禁处停止。

## 前置条件

- 确认用户明确请求了 Debug Mode。否则只能提醒该工作流可用。
- 阅读目标仓库规则以及相关源码、调用链和数据流。
- 收集预期与实际行为、复现步骤、错误以及现有 trace 或失败测试。使用已能得出结论的证据；只对仍未解决的问题添加仪表。

## 仪表契约

- 将调试输出写入文件，绝不使用 `print`、`debugPrint`、`console.log`、stdout 或 stderr。
- 优先使用 `{project_root}/.agents/debug.log`。清空或删除前，确认该文件属于本次 debug session；否则在 `.agents/` 下选择任务专用文件并报告其路径。
- 添加仪表前只解析一次绝对日志路径。不要在运行时从进程工作目录推导路径。
- 对浏览器代码，优先使用现有本地调试 endpoint。确实需要临时 endpoint 时，将其限制为仅本地可用并在清理阶段删除；不要在生产构建中暴露调试日志。
- 每条记录以假设编号开头，例如 `[DEBUG H2]`，并且只记录评估该假设所需的状态、时序、分支或生命周期事件。
- 将每项临时修改包装在语言适用的 region 中：

```text
// #region DEBUG       Dart、JavaScript、TypeScript、Java、C#、Go、Rust、C、C++
# #region DEBUG        Python、Ruby、Shell、YAML
<!-- #region DEBUG --> HTML、Vue、Svelte
-- #region DEBUG       Lua、SQL
```

使用该语言匹配的 `#endregion DEBUG` 形式。

## 阶段 1：定义问题

说明预期行为、实际行为、已知最短复现和相关执行路径。只询问缺少且会阻止形成可检验假设的信息。

## 阶段 2：形成假设

列出少量编号、合理且可区分的假设。有证据支持时包括非显而易见的时序、状态、所有权、初始化和生命周期原因。对每个假设说明什么观察会确认或排除它。

## 阶段 3：添加仪表并等待

1. 确认日志文件所有权，并在复现前清空它。
2. 为当前假设添加最少的 region 包装仪表。
3. 告诉用户准确的复现方法和将写入的日志文件。
4. 停止并等待用户复现 bug。

## 阶段 4：诊断

1. 读取内容前先检查日志大小。文件较大时使用平台原生筛选提取假设记录。
2. 将每个假设标记为 `confirmed`、`ruled out` 或 `unresolved`，并引用日志证据。
3. 只有证据能区分根因和其他可能时才陈述根因。
4. 如果仍未解决，修订假设、清空本次 session 自有日志、调整仪表，并返回复现门禁。

## 阶段 5：修复并等待

保留仪表并实现最窄的根因修复。清空本次 session 自有日志，请用户验证原始复现，然后停止并等待。

## 阶段 6：清理

用户确认修复后：

1. 删除所有 `DEBUG` region 和任何临时 endpoint。
2. 只删除本次 debug session 自有的日志文件。
3. 运行相关项目检查，并总结根因、证据、修复和验证。

如果修复失败，保留仪表和恢复证据，读取新日志，并返回阶段 2。

## 停止条件

- 在证据能区分可疑原因前，不要修补它。
- 没有用户复现结果时，不要越过任一复现请求。
- 不要清空或删除不属于本次 session 的既有文件。
- 用户验证修复前，不要删除仪表。
