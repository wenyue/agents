# 调试模式

修复 bug 之前，先通过可验证的假设、有针对性的文件插桩和人工复现完成诊断。严格按阶段顺序执行，并在每个复现关口停下来等待。

## 前提条件

- 确认用户明确要求使用 Debug Mode。否则只能告知用户这一流程可用。
- 阅读目标仓库的规则，以及相关源代码、调用链和数据流。
- 收集预期行为、实际行为、复现步骤、错误，以及已有的跟踪记录或失败测试。已有证据足以得出结论时直接采用；只为尚未查明的问题添加插桩。

## 插桩契约

- 调试输出必须写入文件，绝不能写入 `print`、`debugPrint`、`console.log`、stdout 或 stderr。
- 优先使用 `{project_root}/.agents/debug.log`。清空或删除前，先确认该文件归本次调试会话所有；否则在 `.agents/` 下选择一个任务专用文件，并报告其路径。
- 插桩前只解析一次日志的绝对路径。不得在运行时根据进程的工作目录推导该路径。
- 对浏览器代码，优先使用已有的本地调试端点。如果确实需要临时端点，只允许本地访问，并在清理阶段移除；不得让生产构建暴露调试日志功能。
- 每条记录都以前缀标明假设编号，例如 `[DEBUG H2]`，并且只记录判断该假设所需的状态、时序、分支或生命周期事件。
- 每项临时改动都要用对应语言适用的 region 包裹：

```text
// #region DEBUG       Dart, JavaScript, TypeScript, Java, C#, Go, Rust, C, C++
# #region DEBUG        Python, Ruby, Shell, YAML
<!-- #region DEBUG --> HTML, Vue, Svelte
-- #region DEBUG       Lua, SQL
```

结束 region 时，使用该语言对应的 `#endregion DEBUG` 形式。

## 阶段 1：界定问题

说明预期行为、实际行为、目前已知的最短复现步骤和相关执行路径。只有缺少的信息会阻碍形成可验证假设时才询问。

## 阶段 2：提出假设

列出少量带编号、合理且能够相互区分的假设。证据支持时，应考虑不易察觉的时序、状态、所有权、初始化和生命周期问题。对每项假设说明何种观察结果能够确认或排除它。

## 阶段 3：插桩并等待

1. 确认日志文件的所有权，并在复现前清空它。
2. 只为当前假设添加必要的最少插桩，并用 region 包裹。
3. 明确告诉用户如何复现，以及数据会写入哪个日志文件。
4. 停止，等待用户复现 bug。

## 阶段 4：诊断

1. 读取内容前先检查日志大小。文件较大时，使用平台原生筛选功能提取与假设相关的记录。
2. 将每项假设标记为 `confirmed`、`ruled out` 或 `unresolved`，并引用对应的日志证据。
3. 只有证据足以将根因与其他可能区分开时，才能认定根因。
4. 如果仍无结论，调整假设、清空本次会话拥有的日志、修改插桩，再回到复现关口。

## 阶段 5：修复并等待

保留插桩，同时针对根因实施范围最小的修复。清空本次会话拥有的日志，请用户按原步骤验证，然后停止并等待。

## 阶段 6：清理

用户确认修复后：

1. 移除所有 `DEBUG` region 和临时端点。
2. 只删除本次调试会话拥有的日志文件。
3. 运行相关项目检查，并总结根因、证据、修复和验证结果。

如果修复失败，保留插桩和恢复证据，读取新日志，然后返回阶段 2。

## 停止条件

- 在证据能够将疑似原因与其他可能区分开之前，不得据此打补丁。
- 在两次要求复现的任一关口，都不得在没有用户结果的情况下继续。
- 不得清空或删除不归本次会话所有的既有文件。
- 用户验证修复之前，不得移除插桩。
