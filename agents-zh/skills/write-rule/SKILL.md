# 编写 Rule

创建一个连贯的 Rule 契约，使另一个 Agent 无需隐藏上下文即可应用。编写前先确定其分发范围和契约
角色；每次调用都对整个 Rule 的质量负责，即使请求只指出一个缺陷。

## 整份 Rule 重写不变量

- 每次创建或更新都从 Rule 当前的完整目标出发重新综合。
- 把现有 Rule 及其发现 surface 视为证据和遗漏检查表，而不是不可变骨架或首选大纲。
- 在决定保留哪些现有表述前，重新建立分发范围、所有权、`Strength`、`Scope`、优先级、边界、
  异常和发现 surface。
- 生成完整候选并连贯替换拥有的产物。Diff 只是交付机制；绝不要为了最小文本补丁而优化 Rule。
- 把每项有效要求整合到其自然 owner。删除已取代、重复、矛盾、错位以及记录修改历史的内容。
- 不要仅为处理最新请求而追加异常、备注、附录或尾部章节。重写控制该行为的章节，然后再次审查整个
  Rule。
- 当当前意图和证据仍支持时，保留 Rule 的既定方向、技术选择、阈值、`Strength` 和有效异常；
  但不要仅因薄弱结构或陈旧表述已经存在就保留它。

## 编写前分类

先确定所有权范围。共享产物分为两类：直接应用自身政策的共享 Rule，以及编写完整目标自有 Rule 的
共享 Rule 生成契约。使用 manifest、catalog、同步所有权、安装范围、wrapper 生成和请求输出作为
证据。

| 类别 | 目标和证据 | 允许的具体程度 | 必需边界 |
| --- | --- | --- | --- |
| 项目私有 Rule | 服务于一个目标仓库并由其拥有，无论手写还是生成 | 当前仓库根相对路径、真实 API、命令、module、lint、服务和项目政策 | 优先保证目标准确性；不要发布或泛化不受支持的事实 |
| 共享 Rule | 跨仓库分发并直接应用自身政策 | 跨仓库不变量和稳定的协议自有路径 | 应用时解析项目事实，并把目标专用政策交给项目私有 Rule |
| 共享 Rule 生成契约 | 跨仓库分发，根据目标证据编写完整的目标自有 Rule | 语义化证据类别和稳定的协议自有路径 | 描述编写过程和生成契约；不要表现得像最终目标 Rule |

项目私有 Rule 通常包含其仓库的最终政策。如果明确需要项目私有 Rule 生成契约，使用生成契约结构，
但证据和输出仍由该仓库拥有。不要仅通过删除具体路径把目标专用 Rule 变成共享 Rule；应围绕真正稳定
的跨仓库政策重新设计，否则保持私有。

## 证据

读取拥有的完整 Rule family，并收集适合其类别的证据。

共同证据：

- 用户要求的结果、示例、约束、技术选择、阈值和兼容要求；
- 现有 Rule、更宽泛和更具体的 Rule、wrapper、索引、镜像、manifest、catalog、validator、测试和
  真实使用产物；
- 适用的优先级、编号、事实源、分发和平台加载政策。

类别专用证据：

- 对项目私有 Rule，检查当前项目 Rule、代码、配置、schema、API、命令、lint、服务、生成文件
  所有权、module 边界和验证行为。这些事实优先于通用最佳实践。
- 对共享 Rule，识别在受支持仓库和平台中持续正确的最小政策。把单一仓库事实视为示例而非默认值。
- 对共享 Rule 生成契约，检查其生成 Rule 必须消费的目标证据类别，并使用存在实质差异的代表性仓库
  测试契约。

把稳定行为要求与实现历史、先前编辑、个人偏好以及应属于 Skill 的多步骤流程分开。

## 路径和所有权规则

- 绝不要在 tracked Rule 或 wrapper 中硬编码绝对文件系统路径。不得嵌入盘符、用户主目录、
  checkout 位置或机器专用临时目录。
- 在项目私有 Rule 中使用仓库根相对路径；只有当前项目证据证明时，才记录具体 API、命令、module、
  服务和 lint。
- 在共享 Rule 中使用语义化项目目标和稳定的协议自有路径。只有共享协议在受支持目标中拥有并标准化
  某路径时，才能保留该具体路径。
- 在共享 Rule 生成契约中，除非路径属于共享生成协议本身，否则用语义方式描述目标文件。要求生成的
  项目 Rule 从目标证据解析具体路径和政策。
- 通用 Rule 编写约束属于 `write-rule`。目标事实和行为政策属于目标仓库的项目私有 Rule；应引用
  而不是复制。
- 稳定的常驻约束属于 Rule。可复用的多步骤流程属于 Skill，运行时事实属于拥有它的配置或工具 surface。
- 在接受请求指定的目标文件前确定自然 owner。用户指定的文件说明期望 surface，但不能证明政策属于
  那里。意图明确时，在自然 owner 中实现政策并报告目标文件冲突；只有移动政策会实质改变请求行为或
  超出当前权限时才询问。
- 每项指令只保留一个事实源。除非平台需要额外运行时字段，wrapper 只包含平台元数据或事实源引用。

## 必需格式和契约结构

每个 Rule 都应具备：

1. 一个命名所拥有政策的 H1 标题。
2. 使用目标规则体系支持的词汇和优先级声明 `Strength`。
3. 一句说明所拥有职责及其重要边界的 `Scope`。
4. 与目标发现体系一致的编号、文件名、glob 或适用性声明。
5. 祈使式指令、明确所有权、基于证据的异常、验证，以及与更宽泛和更具体 Rule 的已定义关系。

目标使用本仓库 Rule 格式时，保留以下开头结构：

```markdown
# Rule Title

Strength: `Mandatory|Default|Advisory`

Scope: One precise sentence naming the rule's owned behavior.
```

分类后选择契约结构。只有职责证明必要时才增删章节；不要混合多套完整模板。

| 契约结构 | 必需章节顺序 | 额外约束 |
| --- | --- | --- |
| 项目私有直接 Rule | H1；Strength；Scope；按职责排序的政策章节；必要时添加 Boundaries 或 Exceptions | 使用已验证目标事实、目标编号和目标规范词汇 |
| 共享直接 Rule | H1；Strength；Scope；按职责排序的政策章节；必要时添加 Boundaries 或 Precedence | 只保留跨仓库政策，并把目标 override 留给项目私有 Rule |
| 共享 Rule 生成契约 | Generation Contract；Evidence；Content；Boundaries | 生成完整的目标自有候选；绝不混合编写指令和最终目标政策 |

## 工作流

1. 把产物分类为项目私有 Rule、共享 Rule 或共享 Rule 生成契约。起草前记录该决策的证据。
2. 定义 Rule 必须拥有的完整政策和必须排除的职责。从证据推断清晰细节；只有未解决的歧义会实质改变
   行为时才询问。
3. 盘点每项现有声明和发现 surface。将其分类为保留、重写、删除、新增或移动到实际 owner。
4. 从头陈述 Rule 的分发范围、所有权、`Strength`、`Scope`、优先级、不变量、异常和跨 Rule
   边界，再考虑复用当前标题或句子。
5. 选择契约结构，并按必需顺序勾勒完整候选。
6. 为精确适用性和优先级编写开头 metadata。除非当前证据要求有意改变，否则保留目标编号和规范词汇。
7. 把正文写成一个连贯的政策契约。将保留事实整合到自然章节，而不是保留旧文档顺序。
8. 在同一连贯变更中更新所有拥有的 wrapper、索引、语言镜像、manifest、catalog 和契约测试。保留
   无关项目自有文件和既有 Git 状态。
9. 按优先级、具体程度、强制约束、当前证据和明确用户意图解决冲突。绝不要把冲突版本留在不同章节。
10. 不查看 diff，直接通读完整候选并应用防退化和验证门禁。任一门禁失败时再次重写。

## 内容和边界

- 只包含另一个有能力的 Agent 为正确实现或 review 目标所需的稳定政策、所有权约束和基于证据的决策。
- 设计一个连贯职责。把无关政策拆分到自然 owner，但不要把一个不变量拆散到必须总是一起读取的多个
  Rule。
- 让具体程度匹配所有权：项目私有 Rule 使用具体仓库事实，共享 Rule 使用稳定跨仓库政策，生成契约
  使用语义化证据类别。
- 每个 bullet 表达一项义务。适用范围不明显时，把触发条件、要求行为和异常写在一起。
- 使用具体、可验证的语言和目标规范词汇。避免无证据的最佳实践、口号、个人偏好和不增加可执行含义
  的理由。
- 仅在能消除行为歧义时使用简洁示例。不要让示例引入规范正文中不存在的政策。
- Wrapper 保持精简；只有适用性、分发或验证需要时才更新发现 surface。
- 语言镜像应保留路径、命令、标识符、代码块、分类、`Strength` 和行为含义。

## 防退化门禁

出现任一条件时拒绝候选并重新编写：

- 最新请求以追加备注出现，而不是改变控制该行为的政策；
- 同一义务在多个章节重复，或与旧异常冲突；
- 项目私有事实泄漏进共享契约，或本地 Rule 以可移植性为名退化成模糊通用建议；
- 共享 Rule 生成契约包含最终目标政策，或共享 Rule 包含另一个 Rule 的编写指令；
- 宽泛 Rule 重复更具体 Rule 或意外 override 它；
- Rule 包含应属于 Skill 的可复用多步骤 workflow；
- 仍保留过时名称、路径、假设、占位符、wrapper 或废弃政策；
- 文档增长却没有新增独立职责或必要证据；
- 旧章节顺序仅因要缩小 diff 而保留；
- 内容描述变更历史、review 讨论或最近修复了什么。

验收前，必须对以下所有问题回答“是”：

- 如果不存在旧版本，今天是否仍会写出这个 Rule？
- 其项目私有或共享所有权是否明确，并得到 catalog 或仓库证据支持？
- 如果共享，它属于共享 Rule 还是共享 Rule 生成契约是否无歧义？
- 每个段落是否都拥有唯一、当前、可执行的信息？
- Agent 能否识别适用性、优先级、义务、override、异常和交付？
- 除非请求有意改变，既定方向、技术选择、阈值、`Strength` 和有效异常是否都得到保留？
- 除非职责确实扩展，这次重写是否降低或保持了复杂度？

## 验证和交付

1. 运行当前 Rule validator，以及覆盖编号、metadata、wrapper、镜像、manifest、catalog、同步和
   所有权的全部项目契约测试。
2. 对项目私有 Rule，针对目标仓库的真实配置和 enforcement surface，验证其具体路径、命令、API、
   lint、schema、服务和 module 声明。
3. 对共享 Rule，在能覆盖项目 Rule 优先级、本地 override 和适用性的代表性上下文中正向测试应用政策。
4. 对共享 Rule 生成契约，生成或模拟至少一个完整目标候选，并验证目标事实来自证据而非生成器。当
   契约声称广泛可移植时，使用存在实质差异的目标。
5. 结构化比较语言镜像，并保留命令、相对路径、标识符、代码块、分类、`Strength` 和行为含义。
6. 运行仓库格式和 diff 完整性检查，然后再次检查完整最终 Rule 及其发现 surface，而不是只审查修改行。
7. 仍有占位符、绝对路径字面量、不受支持的声明、陈旧发现 surface、失败检查或未解决所有权冲突时，
   不得报告成功。

## 结果

报告产物属于项目私有 Rule、共享 Rule 还是共享 Rule 生成契约，并报告所选契约结构、保留方向、从
全局重写的职责、删除的重复或陈旧内容、拥有的发现 surface、同步镜像和精确验证结果。
