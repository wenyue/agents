# 编写规则

写出一条别人拿来就能执行的完整规则：内容前后一致、有事实依据，不依赖未说明的背景。每次启用时，
都要重新梳理这条规则适用范围内的全部要求，不要被旧版本的结构牵着走。

## 分类

动笔前，先根据规则的分发方式、职责归属和用户要的产物判断类型。

| 判断依据 | 类型 |
| --- | --- |
| 最终规则由一个仓库维护并直接执行 | 项目本地规则 |
| 发布出去的规则会直接约束不同仓库 | 共享规则 |
| 发布出去的内容负责写出一份由目标项目自己维护的完整规则 | 共享规则生成契约 |

删掉本地细节，并不会自动变成共享规则。只有规则本身在不同仓库中都能保持稳定，才把它归为共享规则；
否则继续作为项目本地规则。

## 证据

只查那些可能影响规则内容、归属或验证方式的事实。

### 规则依据

弄清用户真正想要的结果、限制、示例、技术选择、阈值和兼容要求。检查同一职责范围内的整套规则、
适用的更通用规则和更具体规则、当前优先级以及实际用法。

现有规则只用来核对事实和防止遗漏，不要把它当成必须沿用的提纲。稳定的规则要求要和变更记录、
个人偏好、可复用流程，以及本应由其他地方管理的运行时事实分开。

### 仓库事实

如果是项目本地规则，要核对当前代码、配置、schema、命令、服务、lint、生成内容由谁维护、模块边界
和真正执行约束的位置。写进规则的每个具体说法，都要和仓库的实际行为对得上。

### 分发与生成依据

分发或发现方式可能变化时，要检查清单、资产目录、平台适配文件、语言镜像、验证器和测试。如果是共享
规则，要查看有代表性的仓库，找出可以直接套用的最小一组稳定要求，同时核对项目本地规则的优先级和
例外写法。如果是共享规则生成契约，要先确定所有目标项目都必须收集哪些证据，再检查有代表性的目标
规则体系、优先级机制、生成入口和验证器。

## 编写

1. 先说清楚这条规则完整负责哪些要求、哪些事情不归它管，以及凭什么判断它的类型、强度和适用范围。
2. 按当前用户意图和证据重新整理出一份完整版本。保留用户想要的结果和已经核实的决定，包括阈值、
   技术选择和例外；删掉过时、重复、冲突或放错位置的内容。假设今天从零开始写这条规则，按最合理的
   顺序组织它；每项要求都放到应在的位置，不要为了缩小 diff 而在末尾补一句，也不要机械保留旧顺序。
3. 每项要求都写进负责它的 Rule。用户点名的 Rule 不负责某项要求时，不要写进去。只有负责该要求的
   Rule 已经在用户请求范围内时，才能直接修改；否则先取得用户明确同意。
4. 只修改 Rule，以及让这些 Rule 正确加载或分发所必需的平台适配文件、索引、清单、语言镜像和契约
   测试。其他类型的改动都超出范围，必须报告。
5. 不看 diff，从头读一遍完整成稿。不断精简和调整，直到每一段都提供不重复、仍然有效而且可以强制
   执行的信息。

每项要求都只能有一种明确解读。约束对象、适用范围、条件、强度和预期行为等信息，只要会影响含义，
就必须写明。除非有新证据或用户明确同意，不得扩大、缩小、弱化、强化或改写原意。

重点写清目标状态、可以观察到的条件，以及谁负责什么，不要把篇幅浪费在过程复述上。如果有多种做法
都合理，就给执行 Agent 留出选择空间；如果顺序会影响正确性或安全性，就明确规定步骤。示例只在确实
需要消除歧义时使用，并且尽量简短。

纳入版本控制的规则只能写仓库相对路径或协议规定的稳定路径，不能写某台机器专用的绝对路径。每项要求
只能在一个地方维护；平台适配文件只保留平台必需的元数据，再加一个指向源文件的引用。

提到其他 Rule 或 Skill 时，使用目标系统声明或认可的正式名称，不要写它在文件系统里的路径。

根据规则类型确定内容边界：

- 项目本地规则要根据已经核实的仓库事实写清最终要求，并把相关要求、相互关系和例外放进负责它们的
  最具体规则；
- 共享规则只写跨仓库都成立的约束、不绑定具体实现的目标条件和协议规定的路径，具体实现和更窄的
  例外留给项目本地规则；
- 共享规则生成契约要把编写说明和最终规则分开，只规定目标项目要查哪些证据，不能凭空编造目标事实。

## 规则契约

### 基本格式

每条规则都必须这样开头：

```markdown
# Rule Title

Strength: `Mandatory|Default|Advisory`

Scope: One sentence naming the rule's owned responsibility.
```

只有规则负责的内容确实需要时才增加章节，不要为了套通用模板而凑章节。

### 规则正文

范围更广的规则不能重复更具体规则的内容，也不能在不明说的情况下覆盖它。

根据适用范围组织正文：

- 项目本地规则必须包含 H1、Strength、Scope 和按职责组织的要求；需要表达已经核实的规则关系、职责
  边界或例外时，再添加 Boundaries 或 Exceptions，不要为了凑结构添加空章节；
- 共享规则必须包含 H1、Strength、Scope 和稳定约束；需要时添加 Boundaries 或 Precedence，并明确
  项目本地规则的优先级，但不要重复目标项目特有的要求。

### 生成契约

共享规则生成契约必须包含 Generation Contract、Evidence、Content 和 Boundaries。它要产出一份完整、
由目标项目自己维护的候选规则，不能把编写流程和最终项目要求混在一起。

## 验证

### 成品检查

- 从头阅读最终规则，不要只看改动行；确认分类、归属、强度、Scope、适用范围、优先级、例外，以及
  这条规则负责的全部内容都清楚且一致。
- 确认改动只限于 Rule 及其管理的文件和入口。
- 检查最终成稿是否存在歧义或语义偏移：每项要求必须只有一种明确解读；只有出现新证据或用户明确同意时，
  才能改变原意。任一条件不满足都不能通过。
- 按职责逐个检查子节。适用于所有类型的要求要留在公共内容里；有条件的要求要放在真正负责它的主题
  下并写明适用条件；不要只为了结构对称而给每种类型分别开子节。
- 对照语言镜像的结构，确保路径、命令、标识符、代码块、分类、强度和行为没有走样。

### 适用环境检查

- 项目本地规则要在当前仓库里核对具体说法、执行位置、例外和规则之间的关系；
- 共享规则要选择有代表性的环境测试，覆盖项目本地规则的优先级、允许的覆盖方式，以及不依赖单个
  仓库细节的稳定要求；
- 共享规则生成契约要至少模拟生成一份完整的目标规则，并核对它的证据、内容和边界；如果声称能广泛
  移植，还要选择差异明显的多个目标。

### 加载入口检查

对每个必需的发现入口运行当前项目规定的验证器、契约测试和 diff 完整性检查。确认适用的平台适配文件、
索引、清单、语言镜像和其他分发入口仍然一致。

只要证据还有疑点、说法没有事实支持、必需的发现入口已经过时或无法访问，或者必需检查失败或没有报告，
就不能声称工作成功。

## 结果

交付时说明：产物属于哪一类、由哪个产物或仓库负责、最终文档结构、保留了哪些决定、删除或迁移了哪些
内容、更新了哪些发现入口、分发入口和语言镜像，以及各项验证的具体结果。
