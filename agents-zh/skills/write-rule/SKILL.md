# 编写规则

写出一份无需额外背景也能直接执行的完整规则。动笔前先判断它如何分发、承担什么职责；
无论用户只指出一个问题还是要求整体重写，都要对整份规则的质量负责。

## 整体重写原则

- 每次创建或更新规则，都要从它现在应当承担的完整职责出发，重新组织全文。
- 现有规则及其发现入口只是证据和查漏清单，不是必须沿用的框架或大纲。
- 在决定保留哪些旧表述前，先重新确认分发方式、所有权、`Strength`、`Scope`、
  优先级、边界、例外和发现入口。
- 先形成完整的新版本，再整体替换归自己负责的文件。差异只是交付手段，不要为了缩小 diff
  而牺牲规则质量。
- 把每项有效要求放到真正负责它的位置，删除已经失效、重复、矛盾、放错位置或只是在记录修改经过的内容。
- 不要为了回应这次请求，在末尾补一段例外、说明、附录或补丁式章节。应直接改写真正约束该问题的章节，
  然后重新审阅全文。
- 只要当前意图和证据仍然支持，就保留规则既定方向、技术选择、阈值、`Strength` 和有效例外；
  但不要因为某种结构或说法已经存在，就把薄弱结构和过时表述一起留下。

## 写之前先分类

先判断文件归谁所有。共享文件要进一步区分：它是直接执行政策的共享规则，还是用来编写目标项目规则的
共享规则生成契约。判断时以清单、目录、同步归属、安装范围、包装文件生成方式和用户要求的产物为证据。

| 类型 | 用途和判断依据 | 可以写到多具体 | 必须守住的边界 |
| --- | --- | --- | --- |
| 项目本地规则 | 服务于一个目标仓库，由该仓库负责；手写或生成都一样 | 可以写当前仓库里的相对路径、真实 API、命令、模块、lint、服务和项目政策 | 以目标项目的准确性为先；不要发布或泛化未经证实的事实 |
| 共享规则 | 分发到多个仓库，直接约束行为 | 只写跨仓库都成立的不变量，以及由共享协议统一规定的稳定路径 | 应用时再解析项目事实，具体政策交给项目本地规则 |
| 共享规则生成契约 | 分发到多个仓库，用目标项目证据编写一份完整的目标规则 | 只写语义层面的证据类别，以及共享协议统一规定的稳定路径 | 描述如何编写和目标规则应具备什么；不要冒充最终规则 |

项目本地规则通常直接写明该仓库最终执行的政策。只有明确需要“项目本地规则生成契约”时，
才采用生成契约的写法，而且证据和产物仍归该仓库所有。不能只删掉具体路径，就把目标项目专用规则
包装成共享规则；要么围绕真正跨仓库稳定的政策重新设计，要么继续让它留在本地。

## 证据

完整阅读归该规则负责的整个规则族，并按文件类型收集证据。

通用证据包括：

- 用户要求的结果、例子、限制、技术选择、阈值和兼容性要求；
- 现有规则、更宽泛和更具体的规则、包装文件、索引、语言镜像、清单、目录、验证器、测试和真实使用文件；
- 适用的优先级、编号、唯一来源、分发方式和平台加载政策。

不同类型还要补充以下证据：

- 项目本地规则：检查当前项目规则、代码、配置、schema、API、命令、lint、服务、生成文件归属、
  模块边界和验证方式。项目事实优先于通用最佳实践。
- 共享规则：找出在所有支持的仓库和平台中都成立的最小政策。单个仓库的事实只能作为例子，不能当成默认值。
- 共享规则生成契约：检查目标规则需要读取哪些目标证据，并用差异足够大的代表性仓库检验这份契约。

把稳定的行为要求，与实现经过、旧改动、个人偏好以及本应放进技能的多步骤流程分开。

## 路径与所有权

- 跟踪进仓库的规则或包装文件不得写死绝对路径。不要写盘符、用户目录、检出目录或机器专用的临时目录。
- 项目本地规则使用相对仓库根目录的路径；只有在当前项目证据确实证明后，才能写入具体 API、命令、
  模块、服务和 lint。
- 共享规则应使用语义化的项目目标和协议统一的稳定路径。只有共享协议在所有支持目标上都明确规定了某条路径，
  才能保留具体路径。
- 共享规则生成契约应从语义上描述目标文件；只有共享生成协议本身拥有的路径才可写死。
  生成出的项目规则必须根据目标证据解析自己的具体路径和政策。
- 所有规则都适用的编写约束放在 `write-rule`；目标项目事实和行为政策放进对应项目的本地规则，
  其他地方引用它，不要复制一份。
- 长期生效、始终适用的限制放在规则里；可复用的多步骤流程放进技能；运行时事实留在拥有它的配置或工具界面。
- 先确定要求真正归谁负责，再处理用户点名的文件。用户给出的文件只是希望看到变化的界面，不能证明政策就属于那里。
  如果意图明确，应把政策写进真正的责任方，并说明点名位置与真实归属不一致；只有迁移会实质改变行为或超出当前权限时才提问。
- 每条指令只保留一个唯一来源。除非平台确实要求额外运行字段，包装文件只能包含平台元数据或源文件引用。

## 必备格式与契约结构

每份规则都要包含：

1. 一个能准确说明所管政策的一级标题。
2. 使用目标规则系统所支持词汇和优先级的 `Strength` 声明。
3. 一句明确责任及关键边界的 `Scope`。
4. 与目标发现机制一致的编号、文件名、glob 或适用条件声明。
5. 命令式要求、明确所有权、有证据支撑的例外、验证方式，以及它与更宽泛或更具体规则的关系。

目标采用本仓库的规则格式时，保留以下开头：

```markdown
# Rule Title

Strength: `Mandatory|Default|Advisory`

Scope: One precise sentence naming the rule's owned behavior.
```

分类后再选择契约结构。某一职责确实需要时才增加章节，不要把多个完整模板拼在一起。

| 契约结构 | 章节顺序 | 额外要求 |
| --- | --- | --- |
| 项目本地直接规则 | 一级标题；Strength；Scope；按职责组织的政策章节；必要时增加 Boundaries 或 Exceptions | 使用已验证的目标项目事实、编号和规范用词 |
| 共享直接规则 | 一级标题；Strength；Scope；按职责组织的政策章节；必要时增加 Boundaries 或 Precedence | 只保留跨仓库政策，目标项目的覆盖项交给本地规则 |
| 共享规则生成契约 | Generation Contract；Evidence；Content；Boundaries | 生成完整的目标项目规则草案；不要把编写说明和最终目标政策混在一起 |

## 工作流程

1. 判断文件属于项目本地规则、共享规则还是共享规则生成契约。起草前先记下支持这一判断的证据。
2. 明确这份规则应完整负责什么、明确不负责什么。能从证据确定的细节直接确定；
   只有未解决的歧义会实质改变行为时才向用户提问。
3. 盘点现有每项声明和发现入口，逐项标记为保留、重写、删除、新增，或迁移到真正的责任方。
4. 先从头确认分发、所有权、`Strength`、`Scope`、优先级、不变量、例外和跨规则边界，
   再决定是否复用旧标题或句子。
5. 选择契约结构，按规定顺序列出完整草案。
6. 写好开头元数据，准确说明适用范围和优先级。除非当前证据要求有意调整，否则保留目标系统的编号和规范用词。
7. 把正文写成一份前后一致的政策契约。保留的事实应放回自然归属的章节，不要为了照顾旧文而沿用原顺序。
8. 同步更新所有归自己负责的包装文件、索引、语言镜像、清单、目录和契约测试。
   保留无关的项目文件和现有 Git 状态。
9. 按优先级、具体程度、强制约束、当前证据和用户明确意图解决冲突，不能让不同章节各保留一套互相矛盾的说法。
10. 不看 diff，直接通读完整草案，再逐项执行防退化检查和验证。任何一项不通过就继续重写。

## 内容与边界

- 只写稳定政策、所有权限制和有证据支持的决定；内容应恰好足够让另一位有能力的代理正确实现或审查目标。
- 一份规则只承担一组内聚职责。无关政策拆给真正的责任方，但不要把必须一起读取的一条不变量拆散到多份规则里。
- 具体程度要与所有权匹配：项目本地规则写仓库事实，共享规则写稳定的跨仓库政策，
  生成契约写语义层面的证据类别。
- 每个列表项只表达一项义务。范围不明显时，把触发条件、必须执行的行为和例外写在同一项里。
- 使用具体、可验证的语言和目标系统的规范词汇。不要加入没有证据的最佳实践、口号、个人偏好或无法形成约束的理由。
- 只有例子能消除歧义时才使用，而且例子不能偷偷引入正文没有规定的新政策。
- 包装文件保持精简；只有适用范围、分发或验证确实需要时才更新发现入口。
- 语言镜像必须保留路径、命令、标识符、代码块、分类、`Strength` 和原有行为含义。

## 防退化检查

出现以下任一情况，都要推翻并重写当前草案：

- 最新要求只是以附加说明出现，没有改到真正管这件事的政策；
- 同一项义务在多个章节重复，或与旧例外冲突；
- 项目本地事实混进共享契约，或为了所谓通用性把本地规则削弱成空泛建议；
- 共享规则生成契约写入最终目标政策，或共享规则反过来教人如何编写另一份规则；
- 宽泛规则重复了更具体的规则，或无意中覆盖了它；
- 规则包含本应由技能负责的可复用多步骤流程；
- 仍有过时名称、路径、假设、占位符、包装文件或已废弃政策；
- 文档变长了，却没有增加独立职责或必要证据；
- 只是为了减小 diff 而保留旧章节顺序；
- 正文在讲修改历史、审查讨论或最近修了什么。

验收前，以下问题必须全部回答“是”：

- 如果今天从零开始写，这会是最终写出的规则吗？
- 它的项目本地或共享归属是否明确，并得到目录或仓库证据支持？
- 如果是共享文件，能否明确看出它是共享规则还是共享规则生成契约？
- 每段话是否都只承担一项独立、现行、可执行的职责？
- 代理能否看清适用范围、优先级、义务、覆盖项、例外和交付要求？
- 除非用户明确要求改变，既定方向、技术选择、阈值、`Strength` 和有效例外是否都已保留？
- 除非职责确实扩展，整体复杂度是否没有增加，最好还有下降？

## 验证与交付

1. 运行当前规则验证器，以及覆盖编号、元数据、包装文件、语言镜像、清单、目录、同步和所有权的项目契约测试。
2. 对项目本地规则，要根据目标仓库的真实配置和强制执行入口，核对其中的具体路径、命令、API、lint、
   schema、服务和模块声明。
3. 对共享规则，要在有代表性的场景中试用，覆盖项目规则优先级、本地覆盖项和适用条件。
4. 对共享规则生成契约，至少生成或模拟一份完整的目标草案，确认目标事实来自证据而不是生成器臆测。
   如果契约声称适用范围很广，还要选择差异足够大的多个目标验证。
5. 对语言镜像做结构比对，确认命令、相对路径、标识符、代码块、分类、`Strength` 和行为含义都未改变。
6. 运行仓库格式和差异完整性检查。最后再通读一次完整规则及其发现入口，不能只看改动行。
7. 只要还有占位符、绝对路径字面量、无证据声明、过时发现入口、失败检查或未解决的所有权冲突，
   就不能宣告完成。

## 交付结果

说明它属于项目本地规则、共享规则还是共享规则生成契约；同时报告采用的契约结构、保留的既定方向、
整体重写的职责、删掉的重复或过时内容、归自己负责的发现入口、已同步的镜像，以及每项验证的准确结果。
