# 编写规则

写出一条别人拿来就能执行的完整规则：内容前后一致、有事实依据，不依赖未说明的背景。每次启用时，
都要重新梳理它负责的全部要求，不要被旧版本的结构牵着走。

## 分类

动笔前，先根据规则的分发方式和职责归属判断它属于哪一类。

| 类型 | 负责什么 | 需要具体到什么程度 |
| --- | --- | --- |
| 项目本地规则 | 一个仓库最终执行的完整规则 | 写明已经核实的仓库相对路径、命令、API、模块、lint、服务和例外 |
| 共享规则 | 可以直接用于不同仓库的稳定约束 | 只写跨仓库都成立的约束、只表达目标含义而不绑定具体实现的项目目标，以及协议规定的路径 |
| 共享规则生成契约 | 指导生成一份由目标项目自己维护的完整规则 | 规定目标项目要收集哪些证据，以及最终规则必须满足什么契约；不要把目标项目的实际要求写死 |

资产目录、清单、安装范围、同步归属、平台适配文件（wrapper）和用户要的产物，比表面上是否“通用”
更能说明它属于哪一类。删掉本地细节，并不会自动变成共享规则。要么围绕真正稳定的约束重新设计，
要么继续作为项目本地规则。

## 证据

只查那些可能影响规则内容、归属或验证方式的事实：

- 用户真正想要的结果、限制、示例、技术选择、阈值和兼容要求；
- 同一职责范围内的整套规则、适用的更通用规则和更具体规则，以及当前优先级；
- 清单、资产目录、平台适配文件、语言镜像、验证器、测试和实际用法；
- 如果是项目本地规则，核对当前代码、配置、schema、命令、服务、lint、生成内容由谁维护，以及模块
  边界；
- 如果是共享内容，只保留在所有支持的仓库里都成立的最少规则要求和证据类别。

现有规则只用来核对事实和防止遗漏，不要把它当成必须沿用的提纲。稳定的规则要求要和变更记录、
个人偏好、可复用流程，以及本应由其他地方管理的运行时事实分开。

## 编写

1. 先说清楚这条规则完整负责哪些要求、哪些事情不归它管，以及凭什么判断它的类型、强度和适用范围。
2. 按当前需求重新整理出一份完整版本。保留仍有依据的方向、阈值、技术选择和例外，删掉过时、重复、
   冲突或放错位置的内容。
3. 每项要求都放到真正负责它的位置。只有用户点名的文件确实负责这项要求时，才修改该文件；否则应
   修改真正负责这项要求的文件或规则。如果移动要求会明显改变用户原本的请求，先询问用户。
4. 如果适用范围或分发方式需要，同步更新它管理的平台适配文件、索引、清单、语言镜像和契约测试。
5. 不看 diff，从头读一遍完整成稿。不断精简和调整，直到每一段都提供不重复、仍然有效而且可以强制
   执行的信息。

重点写清目标状态、可以观察到的条件，以及谁负责什么，不要把篇幅浪费在过程复述上。如果有多种做法
都合理，就给执行 Agent 留出选择空间；如果执行顺序本身就是规则的一部分，或者实际上没什么合理选择，
就明确规定步骤。示例只在确实需要消除歧义时使用，并且尽量简短。

纳入版本控制的规则只能写仓库相对路径或协议规定的稳定路径，不能写某台机器专用的绝对路径。每项要求
只能在一个地方维护；平台适配文件只保留平台必需的元数据，再加一个指向源文件的引用。

提到其他 Rule 或 Skill 时，使用目标系统声明或认可的正式名称，不要写它在文件系统里的路径。

## 规则契约

如果目标项目采用本仓库的规则格式，开头应写成：

```markdown
# Rule Title

Strength: `Mandatory|Default|Advisory`

Scope: One sentence naming the rule's owned responsibility.
```

标题、强度、Scope、编号或适用范围声明，以及描述优先级的用词，都要采用目标系统支持的格式。只有
规则负责的内容确实需要时才增加章节，不要为了套通用模板而凑章节。

| 契约形式 | 必须包含 | 额外要求 |
| --- | --- | --- |
| 项目本地直接规则 | H1；Strength；Scope；按职责组织的要求；需要时添加 Boundaries 或 Exceptions | 写清已经核实的目标项目要求，以及这些要求之间的关系 |
| 共享直接规则 | H1；Strength；Scope；稳定约束；需要时添加 Boundaries 或 Precedence | 目标项目特有的要求留给项目本地规则 |
| 共享规则生成契约 | Generation Contract；Evidence；Content；Boundaries | 生成一份完整、由目标项目自己维护的候选规则；不要把编写说明和最终项目要求混在一起 |

范围更广的规则不能重复更具体规则的内容，也不能在不明说的情况下覆盖它。

## 验证

- 从头阅读最终规则，不要只看改动行；确认分类、归属、强度、Scope、适用范围、优先级、例外，以及
  这条规则负责的全部内容都清楚且一致。
- 项目本地规则：根据当前仓库实际执行的约束，核对其中的具体说法和规则之间的关系。
- 共享规则：选择有代表性的环境测试，覆盖项目本地规则的优先级和覆盖行为。
- 生成契约：至少模拟生成一份完整的目标规则；如果声称能广泛移植，还要选择差异明显的多个目标。
- 对照语言镜像的结构，确保路径、命令、标识符、代码块、分类、强度和行为没有走样。
- 对所有负责让系统发现这条规则的入口，运行当前项目规定的验证器、契约测试和 diff 完整性检查。

只要证据还有疑点、说法没有事实支持、本应同步维护的入口已经过时，或者必需检查失败或没有报告，
就不能声称工作成功。

## 结果

交付时说明：这条规则属于哪一类、最合适的责任方是谁、最终采用哪种契约形式、保留了哪些决定、删除
或迁移了哪些内容、更新了哪些加载或发现入口和语言镜像，以及各项验证的具体结果。
