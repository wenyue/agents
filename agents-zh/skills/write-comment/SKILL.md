# 编写注释

只有注释能够补充代码、名称、签名、类型或紧邻的控制流尚未表达的信息时，才编写注释。

## 了解项目约定

编写前，阅读适用的仓库规则和语言规则、附近的注释、格式化或 lint 配置，以及生成文件的所有权。以下事项由目标项目决定：

- 注释使用的语言和术语；
- 行注释、块注释和文档注释的语法；
- 标点、大小写、换行和 tag 约定；
- 声明应使用 doc comment、docstring、annotation，还是不加注释；
- `TODO`、抑制指令或 API 文档 tag 等必需标记。

项目约定优先于下述默认规则。

## 决策流程

1. 确定如果没有这条注释，后来的读者会误解什么、违反什么约束，或不得不重新查明什么。
2. 如果答案是“什么都不会”，就不要添加注释；真正的问题在于名称或结构时，应改进名称或结构。
3. 确定注释的作用：API 契约、不变量、生命周期、边界情况、失败行为、设计理由、外部要求或局部意图。
4. 用尽可能短的表述补足缺失的信息。
5. 遵循目标语言的语法，以及项目的语气和格式规则。
6. 将代码与注释放在一起阅读，删去只是复述下一行代码的内容。
7. 运行项目相关的格式化、文档检查、静态分析或 lint。

## 高价值内容

- 调用方无法从签名推断的行为；
- 不变量和执行顺序约束；
- 生命周期、所有权、取消或并发要求；
- 边界情况、fallback 行为和预期失败模式；
- 选择不直观方案的理由，或不采用某个明显替代方案的原因；
- 外部协议、兼容性、安全性或产品要求。

## 项目未作规定时的默认规则

- 与附近仍在维护的文档保持语言和术语一致。
- 公共声明使用该语言原生的文档形式；局部原因说明使用普通行注释或块注释。
- 使用主动、直接的表述，每条注释只表达一个意思。
- 说明性注释使用完整句子；标记、指令、代码片段和 URL 保持所要求的原生形式。
- 使用目标语言支持的文档语法引用参数、异常和符号。

## 示例

好的注释会补充缺失的约束或理由：

```text
// Keep the old token until persistence succeeds so a failed write can be retried.
# The service reports healthy before the index is ready; poll the readiness endpoint instead.
```

不好的注释只是复述代码：

```text
// Increment the retry count.
# Return the cached value.
```

## 保留特殊形式

不得仅为把有效标记、抑制指令、文档 tag、URL、代码片段或生成注释改写成完整句子而修改它们。只有目标项目规则或用户请求要求时，才进行修改。

## 结果

报告在哪些位置添加、修改或有意省略了注释，这些注释保留了哪些不易看出的信息，以及使用了哪项项目检查进行验证。
