# 重构代码

使用此 skill 选择并执行合适的重构深度。

如果用户没有明确选择模式，编辑前先询问使用哪种重构模式：

1. 格式重构
2. 逻辑重构
3. 深度重构

三种模式在修改前都必须遵循当前仓库的 `AGENTS.md` 和适用项目规则。项目规则优先于此 skill。

## 格式重构

使用格式重构改善可读性和局部结构，不改变行为。

约束：

- 不改变逻辑。
- 不改变公共接口。
- 不改变可观察行为、持久化格式、状态语义或跨模块契约。
- 可以拆分或合并函数。
- 名称不属于公共 API 时，可以重命名函数或变量。
- 可以简化局部控制流、调整分支顺序、调整辅助函数边界并减少重复。
- 不要只为减少行数而提取微小辅助函数。

工作流：

1. 识别具体的可读性问题：名称、顺序、分支、重复、辅助函数边界或局部结构。
2. 阅读附近代码，并遵循该区域已有风格。
3. 以保持行为的步骤进行重构。
4. 只有当重构明确使死代码、废弃参数或冗余包装器变得不必要时，才移除它们。
5. 在宣称完成前，使用相关测试、分析器输出或项目专用检查进行验证。

## 逻辑重构

使用逻辑重构从头重建内部逻辑，同时保留外部契约。目标是移除历史负担，使实现更易理解。

约束：

- 不改变公共接口。
- 不改变外部契约、持久化格式或用户可见行为。
- 可以重写内部逻辑、数据流、状态所有权和私有辅助结构。
- 新实现替换旧结构后，不要保留废弃分支、包装器、兼容路径或迁移脚手架；用户明确要求渐进迁移路径时除外。

工作流：

1. 阅读所有需要重构的代码和相关调用位置。
2. 设计替代方案前，先理解当前代码的行为。
3. 向用户询问不清楚的领域规则、边界情况、兼容要求或历史行为。
4. 理解重要细节后，定义目标内部模型：职责、数据流、状态所有权、错误处理和调用方交互。
5. 如果逻辑复杂或风险较高，先添加或更新单元测试，以固定当前输出和边界行为。复杂重构建议使用单元测试，但简单或低风险逻辑不强制要求。
6. 从头重写内部实现，不要在旧结构上叠加新补丁。
7. 如果测试依赖废弃的内部实现，更新测试，使其面向保留的外部行为和新的实现形态。
8. 验证重构前后的外部边界行为保持一致。

## 深度重构

当逻辑重构仍不足以解决问题，并且简化公共接口或外部契约会使设计显著更清晰时，使用深度重构。

约束：

- 只有获得用户确认后，才可以改变公共接口、公共 API、跨模块契约、持久化格式、协议或用户可见行为。
- 改变外部表面前，说明原因、影响、迁移成本和合理替代方案。
- 遇到旧数据、旧协议、旧存储格式、集成或遗留调用方等兼容问题时，询问用户是否要求兼容。
- 明确兼容性取舍：放弃兼容可能显著简化代码，这一成本对项目也可能可以接受。
- 获得确认后，可以删除或简化废弃公共接口、适配器、兼容分支和旧抽象。
- 不要为假设性的未来需求引入通用框架、配置层或扩展点。

工作流：

1. 执行逻辑重构所要求的同等理解、建模和风险分析。
2. 列出应该改变的每个公共接口、外部契约、持久化形态、协议或用户可见行为。
3. 对每个兼容问题，询问用户是保留兼容，还是接受破坏性清理。
4. 编辑这些外部表面前，向用户确认要求的外部变更。
5. 获得确认后，重写实现并更新调用方、测试和文档化契约。
6. 移除用户同意放弃的旧入口和兼容路径。
7. 使用相关测试、分析器输出或运行时检查验证新的公共表面和关键行为。

## 反模式

- 未经确认就把格式重构扩展为逻辑或深度重构。
- 用“重构”掩盖行为、契约、协议或持久化变更。
- 进行大量代码搬动，却没有改善可读性、所有权或逻辑清晰度。
- 保留旧架构不动，只在其外部包装新抽象。
- 仅为方便测试而添加公共 API、测试接缝或适配器。
- 已经没有任何依赖时仍保留旧路径。
