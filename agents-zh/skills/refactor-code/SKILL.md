# 重构代码

选择能够实现所请求结果的最浅重构模式。除非所选模式和用户明确允许改变，否则保留项目规则和可观察契约。

## 选择模式

| 模式 | 使用场景 | 契约变更 |
| --- | --- | --- |
| Format | 改善命名、可读性、局部结构、重复或控制流 | 无 |
| Logic | 替换内部逻辑或状态流，同时保留外部契约 | 仅私有实现 |
| Deep | 简化公共接口、持久化、协议或用户可见行为 | 仅在明确确认后允许 |

请求唯一确定模式时直接推断。只有两个模式会产生实质不同的范围或行为时才询问。

## 共享工作流

1. 阅读适用仓库规则、目标代码、附近模式和受影响调用者。
2. 说明所选模式、行为边界和具体设计问题。
3. 找出现有测试或可观察示例，用于保护要保留的契约。
4. 在所选模式内完成最小连贯重构。
5. 当代码位于批准范围内且没有受支持路径依赖它时，删除新结构淘汰的代码。
6. 报告完成前运行最小相关 formatter、静态检查和测试。

不要为假设的未来需求引入 framework、配置层、测试专用 API 或扩展点。

## Format Refactor

- 保留逻辑、公共接口、持久化格式、状态语义和用户可见行为。
- 只有命名、顺序、分支、重复和 helper 边界能解决已说明的可读性问题时才改善它们。
- 没有具体需要时，不要把局部清理扩大为调用方或模块重组。
- 不要仅为减少行数而提取很小的 helper。

## Logic Refactor

- 保留公共接口、外部契约、持久化格式和可观察行为。
- 替换内部实现前，定义目标职责、数据流、状态所有权、错误处理和调用方交互。
- 重写前用测试保护复杂或高风险的边界行为。
- 替换淘汰的内部分支和 adapter，不要在旧实现外包装第二套架构。
- 更新依赖已退役内部结构的测试，使其断言保留的外部行为。

## Deep Refactor

编辑外部 surface 前：

1. 列出会改变的公共 API、跨模块契约、持久化形状、协议、集成或用户可见行为。
2. 解释收益、影响、兼容成本、迁移选项和更窄的替代方案。
3. 对破坏性范围和兼容策略取得明确确认。

确认后一起更新调用方、测试、迁移和已记录契约。删除用户同意退役的旧 surface 和兼容路径；不要保留休眠的遗留入口。

## 停止条件

- 请求结果需要比已批准模式更深的模式时停止。
- 在未批准的公共、持久化、协议或用户可见变更前停止。
- 将不明确的兼容要求视为决策点，不要默认保留一切或破坏一切。

## 结果

报告所选模式、移除的问题、任何有意契约变更、受影响文件和执行的准确验证。
