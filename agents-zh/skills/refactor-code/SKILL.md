# 重构代码

选择足以达到目标、变更层级最低的重构模式。除非所选模式与用户授权都明确允许改变，否则必须保留项目规则和可观察契约。

## 选择模式

| 模式 | 适用场景 | 契约变更 |
| --- | --- | --- |
| Format | 改进命名、可读性、局部结构、重复代码或控制流 | 无 |
| Logic | 替换内部逻辑或状态流，同时保持对外契约不变 | 仅限私有实现 |
| Deep | 简化公共接口、持久化、协议或用户可见行为 | 仅在明确确认后允许 |

如果请求只能对应一种模式，直接据此判断。只有两种模式会造成实质不同的范围或行为时才询问。

## 通用流程

1. 阅读适用的仓库规则、目标代码、附近的既有模式和受影响的调用方。
2. 说明所选模式、行为边界和需要解决的具体设计问题。
3. 找出保护既有契约的测试或可观察示例。
4. 在所选模式内完成最小且完整的重构。
5. 如果新结构使批准范围内的代码失去用途，且没有受支持的路径依赖它，就将其删除。
6. 报告完成前，运行最少且相关的格式化、静态检查和测试。

不要为设想中的未来需求引入框架、配置层、测试专用 API 或扩展点。

## Format 重构

- 保持逻辑、公共接口、持久化格式、状态语义和用户可见行为不变。
- 只有改动命名、排列、分支、重复代码或辅助函数边界能够解决已指出的可读性问题时，才进行这些调整。
- 没有明确需要时，不得将局部清理扩大为调用方或模块重组。
- 不得只为减少行数而提取微小的辅助函数。

## Logic 重构

- 保持公共接口、外部契约、持久化格式和可观察行为不变。
- 替换内部实现前，先定义目标职责、数据流、状态所有权、错误处理和调用方交互。
- 重写前用测试保护复杂或高风险的边界行为。
- 直接替换过时的内部分支和适配器，不要在旧实现外再包一套架构。
- 更新依赖已移除内部实现的测试，使其改为断言需要保留的外部行为。

## Deep 重构

修改对外部分之前：

1. 列出会发生变化的公共 API、跨模块契约、持久化数据形态、协议、集成或用户可见行为。
2. 说明收益、影响、兼容性成本、迁移方案和范围更小的替代方案。
3. 就破坏性变更的范围和兼容策略取得明确确认。

确认后，同步更新调用方、测试、迁移和文档化契约。删除用户同意停用的旧接口和兼容路径，不得保留闲置的遗留入口。

## 停止条件

- 如果达到请求结果需要比已批准模式更深的模式，立即停止。
- 在尚未批准的公共接口、持久化、协议或用户可见行为发生改变之前停止。
- 兼容性要求不明确时，将其视为必须作出的决策，不得默认保留一切或破坏一切。

## 结果

报告所选模式、已消除的问题、任何有意作出的契约变更、受影响文件，以及实际执行的验证。
