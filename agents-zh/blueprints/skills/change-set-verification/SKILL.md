# 变更集验证

编写一份完整、由目标项目负责的 Skill，在交付前规范化并验证一个内在一致、已经完成的变更集。
本生成契约负责定义 verifier；它不会验证当前变更，也不会实现语义修复。

## 证据

检查目标仓库的 `Project Tools` Rule、更上层和更具体的 Rule、当前实际可调用的工具面、工具配置、
项目清单、软件包边界、测试布局、CI 工作流、生成源文件策略，以及仓库自有的验证选择器。确定：

- 仓库实际支持哪些 formatter、fixer、analyzer、linter、测试、构建、运行时和 diff 完整性检查；
- 每个候选命令的工作目录、前置条件、范围选择、是否修改状态、输出、相对成本，以及与其他工具的重叠；
- 原生工具或 MCP 工具是否保留所需的范围选择与证据，互不依赖的调用能否一起提交，以及长时间调用由哪项
  上层配置负责超时；
- 测试的直接归属、生成源文件责任方、依赖边界，以及有证据支持的“变更到检查”对应关系；
- 哪些风险和工具限制要求扩大验证范围，以及有没有可信方式区分所选失败与目标项目基线；
- verifier 可以机械修复哪些文件和诊断，哪些语义工作必须交还实现责任方。

不要臆造证据无法证明的命令、归属映射或范围支持。

## 编写流程

1. 明确生成 Skill 的触发条件、实现完成检查点前提、所选变更集模型、完成条件、结果状态、停止条件，
   以及不负责的实现工作。
2. 将每类受支持变更映射到足以证明它的最小仓库自有检查。根据依赖、共享契约、生成接口、工具修改、
   归属缺口和风险，定义扩大范围的条件。
3. 安排规范化、只读静态检查、直接归属测试、更广检查和结果分类的顺序；每个独立验证面只有证据仍然
   需要时才运行。
4. 在仓库命令和当前实际可调用的原生或 MCP 工具之间做出选择。analyzer 或测试工具能够保持所需范围和
   证据时优先使用；如果工具会扩大 formatter 或 fixer 的修改范围、丢失必要选择器或削弱诊断，则保留
   仓库命令。
5. 平台支持时，把互不依赖的调用合并到一次编排请求中。合理的长时间操作应在归属工具中配置有证据支持
   的超时；不得改成后台 shell 进程再反复短轮询。
6. 优先复用仓库自有选择器。只有现有工具无法稳定表达反复执行的确定性选择逻辑时，才添加 Skill
   自有脚本；脚本必须遵守生成后项目本地 Skill 自己的运行时规则。
7. 核心决定留在 `SKILL.md`。只有多个软件包、语言或风险映射会妨碍工作流执行时，才添加
   `references/verification-matrix.md`。
8. 不依赖旧版目标 Skill 或 diff，从头阅读完整生成目录。反复调整，直到每条指令、命令、范围决策和
   结果都只有一种明确含义。

## 生成 Skill 契约

### 触发时机与范围

- 只在实现已经完成的检查点、交付之前运行。不得打断仍在进行的编辑、调试或尚未完成的修复循环。
- 根据任务上下文和仓库状态，识别内在一致的目标变更集。保留现有 `HEAD`、index、无关的 staged 或
  unstaged 改动，以及未跟踪文件。
- 找出属于该变更集的生产代码、测试、配置、生成源文件责任方和配套文件。
- 从足以证明结果的最小范围开始。只有依赖关系、共享契约、生成接口、fixer 产生的修改、工具限制或
  归属不明使较小范围不再可信时，才扩大范围。
- 测试归属缺失是需要解决的缺口或扩大范围的理由，绝不能据此认定无需测试。
- 绝不编辑生成产物、第三方代码或所选项目自有范围之外的文件。

### 规范化与修复

1. 项目支持时，格式化所选项目自有源代码范围。
2. 每项获项目认可的自动修复，在其支持的最小选定范围内最多运行一次。如果同一工具已经负责相同的
   analyzer 或 lint 验证面，不要重复执行。可以接受选定范围内的机械修复，但不得只为发现或修复旧问题
   而扩大范围。
3. 将 formatter 或 fixer 修改的所有文件加入所选变更集。如果工具改动了生成文件、第三方文件、
   无关文件或其他非自有文件，立即停止。
4. 必要时重新格式化 fixer 修改的源代码，然后运行受支持的最小范围只读静态检查。
5. 将剩余语义诊断连同准确位置和消息交回实现责任方。verifier 不得自行实现语义修复。
6. 实现责任方修改文件后，将结果视为新的实现完成检查点，并根据当前仓库状态重新开始工作流。

### 工具选择与编排

- 当前可用的原生或 MCP analyzer、测试工具能够遵守同一选定范围、返回等价诊断，并且已配置足够超时时，
  优先于同类 shell 命令。
- 如果对应工具只能修改更大的根目录，继续使用可精确限定范围的仓库 formatter、fixer 命令。工具可用
  不代表可以扩大所选变更集。
- 结果互不依赖的检查应一起提交；规范化、会修改文件的检查，以及依赖生成产物的检查仍须按顺序执行。
- 一次同步原生或 MCP 调用就是一个完整操作，应通过该调用等待结果；不得另起后台进程或反复发送短间隔
  状态查询。

### 验证与结果

- 静态检查通过后，运行直接归属于所选范围的测试。formatter、fixer 或实现责任方修改使组件进入范围时，
  也要运行这些组件的测试。
- 只有证据定义的风险或归属边界确有要求时，才运行更广测试、构建、运行时检查或集成验证面。
- 每个实现完成检查点内，各项不同验证面最多运行一次；只有发生修改且有记录表明确需复查时才重复。
- 将每个选定验证面归类为 `passed`、`failed`、`inconclusive` 或 `not applicable`。报告命令、范围、
  选择理由、结果和剩余缺口。
- 报告所有被修改的文件、formatter 与 fixer 调用、重复检查、剩余诊断和验证缺口。
- 返回一个总体结果：`passed`、`semantic_fix_required`、`failed` 或 `inconclusive`。
- 只要任何必需验证面失败或仍无定论，就不得报告 `passed`。
- 如果范围外失败可能早于当前变更，只将这个失败面与可信基线比较。不得仅为分类而运行完整基线套件。

### 停止与失败处理

- 缺少前置条件、无法安全确定所选范围归属、自动工具改动了禁止范围、无法核对仓库状态，或必需结果
  不可信时，立即停止。保留证据，并根据情况将对应验证面报告为 `inconclusive` 或 `failed`。
- 生成 Skill 如果包含可执行脚本，必须包含 `## Failure Recovery`：报告准确的失败命令和错误，分析原因，
  并在修改或重试脚本前提出一份完整候选变更。
- 不得负责业务实现、语义修复、worktree 创建或集成、依赖安装、Agent 同步和破坏性清理，除非目标仓库
  明确把其中某项定义为所选验证面的一部分。

## 审查关口

实际执行前，从头审查完整生成目录。逐项对照目标证据，确认所有命令、范围映射、修改、去重规则、
扩大范围条件、结果状态、基线规则和交接要求。确认项目 Rule 仍是策略来源、可选资源确有必要且可访问，
并且 verifier 不负责语义实现或 worktree 生命周期。存在无依据或有歧义的映射，就不能通过审查。

## 验收关口

审查通过后，在目标仓库中一个有代表性、内在一致的变更集上实际执行完整生成 Skill。通过真实候选版本
完成正常流程，并核实所选范围、规范化行为、直接归属检查、适用时的扩大范围、各验证面分类、总体结果，
以及无关仓库状态是否保持不变。

还要执行一个安全且相关的停止或失败路径，例如剩余语义诊断、缺少前置条件、禁止范围被修改，或归属
无法确定。记录准确命令、候选版本造成的文件修改、初始与最终仓库状态、诊断、分类和所有未运行项。
如果无法安全执行候选版本，或者无法解释其修改，就不能通过验收。

## 交接

只有两个关口都通过后，才能把完整、通过验收的目录、仓库证据、审查结论、验收证据和未解决或未运行的
验证面交给 `setup-project-agents`。任一关口失败时都应停止并报告阻碍，不能把候选版本当作已经验收的
产物交接。
