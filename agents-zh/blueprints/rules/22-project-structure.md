# 项目结构

强度：`Advisory`

适用范围：生成目标仓库的文件放置边界、模块归属、依赖方向和共享基础设施位置。

## 生成契约

根据观察到的仓库组织方式和经过核实的依赖边界，生成完整、由目标项目负责的
`Project Structure` Rule。只保留能让 Agent 正确放置改动、找到责任方并避免无效依赖的最小职责图，
不要复述目录树。

## 证据

- 结合包清单、模块声明、构建目标和有代表性的 import，检查仓库与工作区目录树。
- 追踪功能代码、共享库、配置、测试、资产、生成源文件、脚本、基础设施和文档实际承担的职责。
- 检查依赖检查、lint 规则、包边界、可见性规则和组合根，再判断某个依赖方向允许还是禁止。
- 只有归属文件和反复出现的放置方式能证明稳定边界时，才把它们作为证据。目录名看起来很直白，本身
  并不能证明存在规则。
- 区分有工具强制的边界和只用于指导放置的模式，并在措辞中体现这种强弱差别。

## 内容

- 只有顶层应用模块、其他区域、软件包或组合根的归属不直观，或选错责任方会影响放置或依赖决策时，
  才描述它。每个选中的条目只写一句简洁的归属说明；不要枚举内部目录或 API。
- 项目已经确立责任方时，写明新功能代码、共享代码、测试、资产、配置、脚本、生成源文件、基础设施和
  文档应放在哪里。
- 写明允许和禁止的依赖方向、适用范围，以及这些方向保护的架构边界。精确层级或排除项应引用权威配置，
  不要复制容易漂移的值。
- 区分真正可复用的共享位置与功能自有 helper，避免为了调用方便而抹掉归属。
- 指明真实的强制机制和权威配置，但具体调用方式留给 `Project Tools`。
- 最终结构图按职责组织并保持简洁；不能帮助判断放置、归属、发现或依赖的位置都应省略。

## 边界

- 工具链、命令、服务、构建和测试调用，以及验证能力归 `Project Tools`。
- API 契约、payload 语义、领域词汇、生成文件编辑策略、生命周期行为和 lint 解释归
  `Project Rules`。
- 不要加入通用架构建议、臆测中的未来结构、临时布局，也不要逐目录罗列。
- 不要在多个章节或 Rule 中重复同一项归属或依赖要求。其他产物才是权威来源时，应引用责任方，不要
  复述其内容。
