# 项目工具

强度：`Mandatory`

适用范围：生成目标仓库的可执行工具、运行时服务、验证入口、工具集成，以及工具与 Skill 之间的
交接规则。

## 生成契约

根据仓库当前证据，生成完整、由目标项目负责的 `Project Tools` Rule。只保留 Agent 必须在行动前
知道的事实和约束：正确选择并不直观、容易误用、重复调查成本高，或做错后返工昂贵。普通且容易发现
的细节仍由对应配置、脚本或命令帮助负责；不要编写工具清单。

## 证据

- 检查包与工作区清单、锁文件、工具链版本配置、仓库脚本、任务运行器、CI 工作流和工具配置。
- 从归属脚本、配置或当前命令帮助中核实关键命令，确认工作目录、前置条件、支持范围、是否修改状态、
  产物和实际成本。
- 检查运行时入口、服务配置、环境模板、凭据边界、端口、启动依赖，以及健康或就绪检查。
- 检查生成器配置和仓库自有的验证、准备选择器，但不要复制生成文件或项目 Skill 负责的策略。
- 检查项目自有的 MCP 和 Agent 平台原生配置。支持多个平台时，对齐名称和用途，同时保留各平台自己的
  schema。
- 缺失、冲突或只在单机成立的证据都应视为未解决，不能写成仓库级事实。

## 内容

- 只有运行时、包管理器、工作区、工具链或工作目录约束会实质影响命令选择或失败方式时，才记录它。
  容易从版本配置或 manifest 安全读取的值应引用责任方，不要复制。
- 对 Agent 必须直接调用的仓库自有目标，记录一小组经过验证的规范命令，例如准备、开发、验证、构建、
  打包或发布。写清所需工作目录，以及不直观的前置条件或范围选择器。
- 其他命令只写归属入口和它负责的决策；只有修改行为、产物或实际成本无法从那里安全判断时，
  才写完整调用。
- 如果混淆只读检查与 formatter、fixer、generator、installer、publisher 等状态修改工具会造成
  明显风险或大范围改动，就明确区分它们。对后果明显的状态修改工具，记录经过验证的范围选择器和
  `dry-run` 模式。
- 只有 Agent 必须连接、维护或验证某项运行时服务、集成、MCP 服务器或原生 Agent 工具面，而且实时
  配置无法直观说明要求时，才将它写入规则。
- 对每个后果明显的生成器，记录其责任方，以及准确命令或可靠的查找入口，同时说明输入和输出。
  不要让 Agent 猜测生成参数，也不要用宽泛更新流程代替狭窄的生成任务。
- 让仓库自有准备、验证入口可以被 `worktree-environment-setup` 或
  `change-set-verification` 直接调用。工作流的触发时机、顺序、扩展范围和结果策略仍由这些
  Skill 负责。
- 只有缺少某项工具或能力会导致 Agent 臆造关键命令或流程时，才明确写出项目没有提供它。

## 边界

- 环境准备过程归 `worktree-environment-setup`，完成变更后的验证过程归
  `change-set-verification`；本规则只向它们提供经过核实的能力和调用约束。
- API 契约、领域行为、生成文件编辑策略和 lint 解释归 `Project Rules`；模块放置与依赖方向归
  `Project Structure`。
- 平台包装文件生成和分发元数据归对应配置或同步清单。
- 不要要求每次任务都运行所有已记录命令，不要重复其他责任方的策略，不要暴露凭据，也不要臆造证据
  无法证明的命令、选择器、成本、服务或集成。
