# 项目工具

强度：`Mandatory`

适用范围：生成目标仓库的可执行工具、运行时服务、验证入口、工具集成，以及工具与 Skill 之间的
交接规则。

## 生成契约

根据仓库当前证据，生成完整、由目标项目负责的 `Project Tools` Rule。只写 Agent 在选择、调用或
维护真实项目工具时必须知道的事实和约束；不要把某台机器上安装的所有工具都列成项目规则。

## 证据

- 检查包与工作区清单、锁文件、工具链版本配置、仓库脚本、任务运行器、CI 工作流和工具配置。
- 从归属脚本、配置或当前命令帮助中核实命令，确认工作目录、前置条件、支持的范围、是否会修改状态、
  产物和成本。
- 检查运行时入口、服务配置、环境模板、凭据边界、端口、数据位置、启动依赖，以及健康或就绪检查。
- 检查生成器配置和仓库自有的验证、准备选择器，但不要复制生成文件或项目 Skill 负责的策略。
- 检查项目自有的 MCP 和 Agent 平台原生配置。支持多个平台时，对齐名称和用途，同时保留各平台自己的
  schema。
- 缺失、冲突或只在单机成立的证据都应视为未解决，不能写成仓库级事实。

## 内容

- 写明会实际影响命令执行的运行时、包管理器、工作区结构、工具链版本和工作目录。
- 按目标组织命令，例如环境准备、开发、生成、格式化、自动修复、分析、lint、测试、构建、打包和发布。
  每条命令都要写清前置条件、适用范围、是否修改状态、产物，以及明显的成本或副作用。
- 区分只读检查与 formatter、fixer、generator、installer、publisher 等会改变状态的工具。
  只有经过核实时，才记录安全的范围选择和 dry-run 能力。
- 说明必需的运行时服务和集成，包括用途、归属配置、启动依赖、端口或 endpoint、环境与凭据要求，
  以及就绪检查。
- 只有项目自有 MCP 服务器和原生 Agent 集成确实属于仓库支持的工具面时，才写入规则。
- 记录生成器入口、源输入和输出。生成文件的语义归属与手工编辑限制留给 `Project Rules`。
- 记录 `worktree-environment-setup` 或 `change-set-verification` 可以调用的仓库自有准备、验证选择器。
  工作流的触发时机、顺序、扩展范围和结果策略仍由这些 Skill 负责。
- 只有缺少某项工具或能力会导致 Agent 臆造关键命令或流程时，才明确记录项目没有提供它。

## 边界

- 环境准备过程归 `worktree-environment-setup`，完成变更后的验证过程归 `change-set-verification`；
  本规则只向它们提供经过核实的能力和调用约束。
- API 契约、领域行为、生成文件编辑策略和 lint 解释归 `Project Rules`；模块放置与依赖方向归
  `Project Structure`。
- 平台包装文件生成和分发元数据归对应配置或同步清单。
- 不要要求每次任务都运行所有已记录命令，不要重复其他责任方的策略，不要暴露凭据，也不要臆造证据
  无法证明的命令、选择器、成本、服务或集成。
