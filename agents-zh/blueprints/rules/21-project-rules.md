# 项目规则

强度：`Default`

适用范围：生成目标仓库的项目专用行为契约、领域约定、生成源文件策略和生命周期不变量。

## 生成契约

根据仓库中稳定的证据，生成完整、由目标项目负责的 `Project Rules` Rule。只保留 Agent 必须在实现前
知道的项目专用行为：普通工具无法稳定发现违规、诊断不能明确指出应如何改正，或发现时已经需要
大范围、高成本返工。保留已经核实的上层规则例外；省略通用指导，以及工具能稳定发现、Agent 可以
局部低成本修复的机械约束。

## 证据

- 检查公共 API、路由、schema、事件、序列化、兼容性测试和真实调用点，找出调用方已经依赖的行为。
- 检查框架配置、项目 analyzer、自定义 lint 和针对性测试，再判断反复出现的写法是否真的是规则，
  或者是否构成对基础 Rule 的例外。
- 从生成产物追溯到源 schema、生成器配置、文件头和重新生成的责任方。
- 检查领域模型、持久化与迁移代码、状态和生命周期责任方、取消、并发及清理行为。
- 检查命名、术语、本地化源和用户可见文案。只有一致用法或强制机制能证明真实项目契约时才采用。
- 证据冲突时，以适用范围最窄的权威来源为准。缺少依据的惯例、历史残留、个人偏好，以及改法明确、
  修复成本低的 lint 细节都应省略，不能写成策略。

## 内容

- 如果违反公共 API、路由、事件、payload、序列化或兼容性约束会越过工具无法稳定保护的使用方或
  兼容性边界，就应写明这些约束。
- 如果普通机械违规的诊断能够直接指出局部、低成本的改法，就交给对应 formatter、analyzer 或
  linter。只有诊断含义会影响架构、归属、合法例外或其他非局部决策时，才在规则中解释。
- 指明生成产物和外部 schema 的语义来源、重新生成义务，以及禁止手工修改的文件或区域。
- 只有领域词汇、命名、标识符、前缀、本地化和用户可见文案会限制有效修改，而且工具无法稳定发现
  违规并直接指出局部、低成本的改法时，才将它们写入规则。
- 项目确有定义时，写明持久化兼容性、迁移、状态归属、生命周期转换、取消、并发和清理不变量。
- 每项对上层 Rule 的例外都要写清准确范围和生效条件，不能在例外范围之外悄悄弱化上层策略。
- 最终规则按负责的行为组织，不要照搬发现证据的先后顺序。

## 边界

- 运行时、工具安装、命令调用、生成命令、服务和验证能力归 `Project Tools`。
- 目录职责、文件放置、模块布局和依赖方向归 `Project Structure`。
- 对生成产物，本规则负责语义来源和编辑边界；`Project Tools` 负责如何调用或找到生成器。
- 不要重复基础 Rule 已覆盖的通用语言风格，不要加入臆测架构、没有文档依据的偏好，也不要重复其他
  权威来源负责的事实。
